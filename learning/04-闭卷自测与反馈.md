# 闭卷自测与反馈

这不是答案稿。它检查你能否讲清五个高回报问题，并从任意代码或 artifact 返回项目主链。

## 使用规则

1. 关闭其他 Markdown 和代码，五题各限时两分钟。
2. 全部答完后再看评分锚点；不要边答边查。
3. 再随机抽一个生产 symbol 和一个 run artifact 做现场定位。
4. 每个错误必须归类，并留下一个命令、代码入口或重答题。
5. 当天、24 小时和 7 天后只复测低于 3 分的题。

| 分数 | 标准 |
| --- | --- |
| 0 | 不会或存在事实错误 |
| 1 | 能认出概念，但没有因果、owner 或边界 |
| 2 | 能解释机制和失败场景，但不能快速给出代码/artifact 证据 |
| 3 | 结论清楚，能指出 owner、状态/副作用、Evidence 和不能声称什么 |

## 闭卷问题

先完成五题，再向下看锚点。

1. **30 秒定位**：NanoHarness 解决什么真实问题？输入、输出、模型/Harness 边界是什么？
2. **黄金主链与架构**：从 `forge run` 到 `RunResult/Run Story` 画出 owner；解释六边形架构的
   编译期依赖和运行时调用为何不同。
3. **Tool Governance**：模型给出一个写文件 ToolCall 后，副作用执行前经过哪些确定性门？
4. **HITL 与 Recovery**：clarification 和 approval 怎样暂停/继续？resume 恢复什么、不恢复什么，
   又怎样防止重复或 stale 副作用？
5. **Evidence**：candidate、local、official 的分母和 claim 有何不同？怎样从一个 artifact 返回
   producer、consumer 和 Run Story？

停在这里。先回答完，再继续。

## 评分锚点

| 题 | 3 分回答必须包含 |
| --- | --- |
| 1 | issue/base commit；隔离 run；candidate patch + evaluation；模型提意图，Harness 管 context/tool/state/safety/eval；不是 IDE/SaaS |
| 2 | CLI 薄适配 -> `Harness.run` 唯一 Public API -> wiring -> AgentLoop -> turn/model -> tool/operation -> lifecycle/checkpoint -> result/story；Application 依赖 Port、运行时调用 injected Adapter |
| 3 | visibility、schema、permission、command/sandbox、approval、operation identity/stale/replay、execute、observation/trace/ledger；prompt 不能授权副作用 |
| 4 | clarification 只补信息、approval 授权具体 operation；durable checkpoint + new continuation；不恢复 stack/KV；operation key、ledger、fingerprint、resume chain |
| 5 | patch/明确 local evidence/parsed official outcome 三种分母；`Unknown` 不等于 0%；artifact producer/consumer/authority/proves/does-not-prove/`rebuildable`/`deletion_impact` |

事实正确但找不到 owner 或 evidence action，最多 2 分。通过标准是至少 13/15、没有 0 分。

## 追问层级

只在五个必会问题达标后练；不要把下面内容塞回主演示。

### A：Core 深挖

- Context 完整预算包含什么？compaction 为什么不能拆 tool transaction？
- Single Agent no-progress 怎样停止？当前语义级重复仍缺什么？
- `RunRequest`、`AgentRunSession`、`TaskCheckpoint`、`RunResult` 生命周期有何不同？
- 为什么 `ToolExecutionPipeline`、`OperationTracker`、`RunLifecycle` 不能直接合进 `AgentLoop`？
- 一个只有单实现的 Port 何时值得保留，何时应合并？

### B：Evaluation 与框架边界

- SWE-bench/Smoke-5 为什么适合机制回归，又不能证明什么？
- 如何设计 matched ablation？为什么当前两个 preset 不能单因素归因？
- 没有 official denominator 时怎样下结论？
- 为什么没有直接使用 LangGraph？生产团队已有 LangGraph 时怎样复用本项目？
- 哪个结果会证伪当前 Coding Agent 定位？

### C：Advanced 能力

- Working Memory、Digest、Long-term Memory 的生命周期和权威性是什么？
- 顺序 Multi-Agent 与 Live Fanout 有何不同，哪些条件 fail closed？
- MCP 和 Skill 如何进入现有 Port/Context/Tool Governance，而不创建第二套 Runtime？
- OCI、worktree、local sandbox 各自提供和不提供什么隔离？

每轮最多选一个层级的一题；答错写入复测记录，不新增回答稿。

## 三个现场动作

### A. 随机 Symbol 定位

在 `10 Production Code` 中随机点开一个 class/public method，限时两分钟回答：层级、规范上游、
下一 owner、状态/副作用、Evidence、删除影响。Call Hierarchy 只能在开始计时后使用。

通过：不是重复选择熟悉入口；所有答案能指向源码或行为测试。若 symbol 没有独立语义，明确提出
内联/降级，不强行为它辩护。

### B. 随机 Artifact 定位

```bash
forge inspect latest
forge inspect <run-or-artifact>
```

限时两分钟指出 producer、consumer、source event、authority、proves/does-not-prove 和可重建性。
从文件名猜测或把 patch 当 solved，均不通过。

### C. 控制面与 Run Story

```bash
forge demo
forge inspect latest
```

指出 request、policy/approval gate、checkpoint、resume、最终副作用和三层 Evidence。主动说明：
deterministic Demo 只证明控制面；真实能力结论来自真实 provider/evaluator artifact。

## 错误分类

| 标签 | 表现 | 补救动作 |
| --- | --- | --- |
| `definition_only` | 只解释名词 | 找失败场景、owner 和 artifact |
| `no_entrypoint` | 说得对但找不到代码 | 对随机 symbol 重走 Caller/Callee |
| `hidden_complexity` | 只会背 `Harness.run`，下游说不清 | 沿黄金主链逐 owner 复述状态变化 |
| `path_confusion` | 混淆 Single/Fanout、clarification/approval | 并排画状态机和触发条件 |
| `evidence_confusion` | 混淆 patch、local、official | 重画 denominator 并定位 artifact authority |
| `abstraction_defense` | 用“规范/扩展性”解释所有类 | 回答真实问题、独立语义和删除影响 |
| `overclaim` | 把 prototype 说成 production/distributed | 重答 claim boundary |
| `implementation_guess` | 不知道时补高级名词 | 标记未知，现场查 owner 后纠正 |
| `slow_answer` | 两分钟仍没有结论 | 使用“结论 -> 两点机制 -> Evidence -> 边界” |

## 已暴露的重点薄弱项

- 把 Harness 只说成“后端”，没有说模型意图和确定性控制层的边界。
- 把 Single Agent 空转误讲成 Fanout DAG 成环。
- 把重复 intent identity 误说成包含尚不存在的执行结果 hash。
- 把 candidate patch、local PASS 与 official resolved 混为同一结论。
- 能背入口但随机打开类/方法后说不出上游、owner 和删除影响。
- 公司经验、个人 clean-room 实现和公开 Evidence 没有先分开。

这些问题没有连续两次达到 3 分前，不新增学习材料或项目功能。
