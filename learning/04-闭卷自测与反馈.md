# 闭卷自测与反馈

这份文档不是答案稿。它用主动提取、代码定位和现场动作检查你是否真正掌握项目。

## 使用规则

1. 关闭其他 Markdown 和代码，只保留问题区。
2. 每题限时口述，录音或写关键词，不允许边答边查。
3. 全部答完后再看评分锚点。
4. 每个错误必须归类，并写一个可执行的补救动作。
5. 24 小时和 7 天后只重测低于 3 分的题。

评分：

| 分数 | 标准 |
| --- | --- |
| 0 | 不会，或说出事实错误 |
| 1 | 能认出概念，但没有因果、入口或边界 |
| 2 | 能解释设计和失败场景，但不能快速给出代码/artifact 证据 |
| 3 | 结论清楚，能说明机制、入口、证据和不能声称什么 |

## 闭卷问题

先完成 15 题，再向下看评分锚点。

1. **30 秒定位**：NanoHarness 是什么？输入、输出和核心问题分别是什么？
2. **模型与 Harness**：模型负责什么，Harness 负责什么？为什么不能只写 system prompt？
3. **运行链**：从 `Harness.run` 到一次工具 observation，按 owner 画出调用链。
4. **四种状态对象**：区分 `RunRequest`、`AgentRunSession`、`TaskCheckpoint` 和 `RunResult`。
5. **No-progress**：Single Agent 一直调用工具但不结束，Runtime 怎样停止？当前没解决什么？
6. **Context**：完整请求预算包含什么？压缩为什么不能拆 tool transaction？
7. **Memory**：working、digest、long-term 三层怎样写入、召回和保持权威？
8. **Tool Safety**：模型返回一个写文件工具后，执行前经过哪些确定性门？
9. **HITL**：clarification、approval、pause、cancel 分别改变什么状态？
10. **Recovery**：resume 恢复什么、不恢复什么？怎样避免重复或 stale 副作用？
11. **Multi-Agent**：顺序 coordinator 与 live fanout 有何不同？冲突怎样处理？
12. **Smoke-5**：候选全集多大、为什么选五题、五题覆盖什么、不能证明什么？
13. **Evidence**：candidate、local verified、official resolved 的分母与 claim 有何不同？
14. **Experiment**：怎样判断 governed-runtime 比 minimal-control 更好？哪些变量必须冻结？
15. **框架选择**：为什么没有直接使用 LangGraph？生产团队已经使用它时怎样复用本项目？

停在这里。先回答完，再继续。

## 评分锚点

评分锚点只列必需事实，不提供可背诵长答案。

| 题 | 3 分回答必须包含 |
| --- | --- |
| 1 | repository issue/base commit；隔离 AgentLoop；candidate patch 与 evaluation evidence；不是 IDE/SaaS |
| 2 | 模型提出意图；Harness 管 context/tool/state/safety/eval；prompt 无法强制副作用边界 |
| 3 | `Harness.run -> build_agent_loop -> AgentLoop.run -> TurnPreparation -> ModelGateway -> ToolExecutionPipeline -> RunLifecycle` |
| 4 | 外部输入、进程内 aggregate、durable continuation summary、外部结果；明确没有 active-task session |
| 5 | step/timeout/cost/exact repeat/consecutive failure/tool burst；Single Loop 没有 DAG；语义 no-progress 仍有限 |
| 6 | instruction、repo context、Skill、memory、messages、tool schema、输出预留；tool call/result 必须原子保留 |
| 7 | current run、continuation digest、evidence-backed cross-run record；candidate -> active -> superseded/retired |
| 8 | routing、schema、permission、command/sandbox、approval、operation/stale、execute、trace/ledger |
| 9 | 补信息、授权具体副作用、协作式暂停、终止 continuation；已执行副作用不自动回滚 |
| 10 | 新 run 的 explicit context；不恢复 stack/KV Cache；operation key、ledger、fingerprint、resume chain |
| 11 | artifact handoff vs isolated DAG worker；declared/actual scope；串行化或 fail closed；不是 swarm |
| 12 | Lite test 300；五仓库/五问题族/F2P+P2P/小 patch；机制回归，不代表总体 solved rate |
| 13 | patch 全体分母、本地 evidence 分母、parsed official denominator；unknown 不等于 0% |
| 14 | matched identity、repetition、paired outcomes、cost/failure；两个 preset 是 multi-factor，不可单因果归因 |
| 15 | 不宣称优于 LangGraph；项目研究 coding-specific execution/evidence；可映射为 node/middleware/checkpointer/service |

事实正确但缺少代码入口或 evidence action，只能得 2 分。

## 四个现场动作

### A. 五分钟代码定位

目标：IDE collapse all 后，不搜索答案文档。

1. 找到 `Harness.run`。
2. 沿一层调用找到 `AgentLoop.run`。
3. 找到工具授权与执行的 owner。
4. 找到 `waiting_human` checkpoint 的写入位置。
5. 说出每层为什么不直接写 JSON 或调用 subprocess。

通过：五分钟内完成，路径与职责均正确。

### B. 控制面演示

```bash
forge showcase hitl start
forge showcase approval start
```

必须能指出 request、checkpoint、continuation command、trace 和最终副作用。还要主动说明：
固定模型行为只让演示可重复，不证明推理质量。

### C. Benchmark 读题

```bash
forge bench cases --regression-set smoke-5
forge bench case astropy__astropy-12907
```

不看说明，现场解释 issue、base commit、F2P/P2P、gold/test patch 隐藏边界，以及该 case
为何适合观察 Harness。

### D. Evidence 诊断

打开一个 run 的 Workbench 或 scorecard，在三分钟内回答：

- 有没有 candidate patch？
- local verification 的分母是什么？
- official outcome 是否存在？
- 当前 failure class 是什么，证据来自哪里？
- 下一次应改 Runtime、模型、环境还是 evaluator？

无法从 artifact 指回结论，视为未通过。

## 错误分类

| 标签 | 表现 | 补救动作 |
| --- | --- | --- |
| `definition_only` | 只会解释名词 | 找一个失败场景和对应 owner 方法 |
| `no_entrypoint` | 说得对但找不到代码 | collapse all 后重走一层调用链 |
| `path_confusion` | 混淆 Single/Fanout、clarification/approval | 并排画状态机，写各自触发条件 |
| `evidence_confusion` | 把 patch、PASS、local、official 混为一谈 | 重新写三层 denominator |
| `overclaim` | 把 prototype 说成 distributed/production | 对照 Capability Reality Matrix 重答 |
| `implementation_guess` | 不知道时补高级名词 | 标记未知，现场查代码后纠正 |
| `slow_answer` | 两分钟仍没给结论 | 强制使用“结论 -> 两点原因 -> 证据 -> 边界” |
| `no_feedback` | 看完感觉懂，没有留下错误 | 必须填写下面的复测记录 |

## 已暴露的重点薄弱项

来自既往模拟面试，优先复测：

- 问 Benchmark 选择时答成 token/tool-call 指标，没有先讲候选全集与采样契约。
- 把 Single Agent 空转误讲成 Fanout DAG 成环。
- 把重复 intent identity 误说成包含执行结果 hash；结果在动作发起前不可用。
- 把 temperature 类比为置信度；它实际控制 sampling 随机性。
- Harness 只答成“后端”，没有说模型与确定性控制层的边界。
- 公司经验、个人 clean-room 实现和公开 evidence 没有先分开。

这些题没有连续两次达到 3 分前，不再新增学习材料。
