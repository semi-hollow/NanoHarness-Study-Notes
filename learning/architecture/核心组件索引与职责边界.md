# NanoHarness 核心组件索引

这是一份按“谁拥有决定”组织的索引。路径只列 canonical implementation；仓库根部的
旧模块多为兼容导出，不应作为新代码入口。

## Runtime 控制面

| 决定 | Owner | 关键协作者 |
| --- | --- | --- |
| 一次 run 的阶段顺序 | `runtime/application/agent_loop.py` | RunPreparation、TurnPreparation、ToolExecutionPipeline |
| 一次 run 的可变字段 | `runtime/application/session.py` | Domain models |
| 输入 guardrail、澄清、Skill | `runtime/application/run_preparation.py` | RunLifecycle |
| Context、tool schema、plan 组装 | `runtime/application/turn_preparation.py` | ContextBuilder、ToolRouter |
| 工具治理顺序 | `runtime/application/tool_execution.py` | Authorization、OperationTracker、Feedback |
| allow/deny/ask | `runtime/application/tool_authorization.py` | HookPort、ApprovalRepository |
| 副作用幂等 | `runtime/application/operation_tracker.py` | OperationLedgerRepository |
| checkpoint、pause、stop 状态 | `runtime/application/run_lifecycle.py` | TaskStateRepository、HumanInputRepository |
| final answer claim 结论 | `runtime/application/final_answer.py` | EvidenceLedger、output guardrail |

## Runtime Ports 与 Adapters

| 外部能力 | Port | Adapter / implementation |
| --- | --- | --- |
| 模型 | `runtime/ports/model.py` | `models/gateway.py`、`runtime/llm_client.py` |
| 工具 | `runtime/ports/tools.py` | `tools/registry.py` 与 concrete tools |
| 事件 | `runtime/ports/events.py` | `observability/adapters/json_trace.py` |
| checkpoint | `TaskStateRepository` | `runtime/adapters/task_state_json.py` |
| 人工问题 | `HumanInputRepository` | `runtime/adapters/human_input_json.py` |
| 审批 | `ApprovalRepository` | `runtime/adapters/approval_json.py` |
| operation ledger | `OperationLedgerRepository` | `runtime/adapters/operation_ledger_json.py` |
| hook | `runtime/ports/hooks.py` | `runtime/hooks.py` |
| 执行环境 | `runtime/ports/environment.py` | `runtime/execution_environment.py` |

具体 Adapter 只由 `runtime/wiring.py` 装配。Application 不知道 JSON、Git、HTTP 或
subprocess 的实现。

## Safety 安全边界

| 组件 | 真实边界 |
| --- | --- |
| `ToolRouter` | 决定当前 turn 对模型可见的 schema，不替代授权 |
| `ToolRegistry` | 工具存在性、参数校验、统一 Observation |
| `PermissionPolicy` | allow/ask/deny |
| `check_command` | shell token、网络和危险命令策略 |
| `WorkspaceSandbox` | workspace path 与 symlink escape |
| `ExecutionEnvironment` | local/worktree/OCI 的运行边界和 manifest |

完整工具链：

```text
Model ToolCall
-> ToolRouter visibility
-> tool guardrail
-> HookManager / PermissionPolicy
-> approval when required
-> ToolRegistry
-> concrete tool
-> WorkspaceSandbox / CommandPolicy / ExecutionEnvironment
-> Observation
```

## Orchestration 编排

| 模式 | Application | Domain | Adapters |
| --- | --- | --- | --- |
| 顺序角色 | `multi_agent/application/coordinator.py` | profile、role、verdict | role runtime、artifact files |
| Live Fanout | `multi_agent/application/live_fanout.py` | FanoutPlan、批次与 conflict rules | local worker、Git workspace、checkpoint files |

两种模式都复用 Runtime 的 `AgentLoop`；Orchestration 不复制工具治理和 HITL 逻辑。

## Benchmark 与 Evaluation

| 责任 | Owner |
| --- | --- |
| case 执行与 artifact 时序 | `bench/application/swebench.py` |
| 失败诊断附加 | `bench/application/diagnostics.py` |
| failure priority | `bench/domain/failure_taxonomy.py` |
| repo/model/official 外部执行 | `bench/adapters/` |
| result card 与 case study | `bench/presentation/` |
| run/variant comparison | `evaluation/domain/comparison.py` |
| scorecard projection | `evaluation/application/scorecard.py` |
| paired ablation | `evaluation/domain/ablation.py` |
| feedback/dataset | `evaluation/adapters/feedback_dataset_files.py` |

Benchmark 完成顺序固定为：执行 case -> staging predictions -> optional official evaluator
-> final diagnosis/case study -> aggregate report。这样 case study 不会比最终评测状态更早落盘。

## Evidence 与 Workbench

| 责任 | Owner |
| --- | --- |
| append-only trace | `observability/adapters/json_trace.py` |
| usage read model | `observability/application/usage.py` |
| usage projection rules | `observability/domain/usage.py` |
| Markdown renderer | `observability/presentation/usage_report.py` |
| local artifact discovery | `workbench/adapters/evidence_files.py` |
| Workbench ports/services | `workbench/ports/`、`workbench/application/services.py` |
| HTTP/HTML view | `workbench/presentation/http.py` |
| bounded browser command | `workbench/presentation/commands.py` |

Trace 保存事实，read model 负责投影，renderer 只呈现；任何一层都不能凭 patch 是否存在
推断 official resolved。

## 推荐阅读顺序

1. `runtime/application/agent_loop.py` 的 `run`
2. `runtime/application/session.py` 字段表
3. `runtime/application/run_preparation.py` 与 `turn_preparation.py`
4. `runtime/application/tool_execution.py` 的两个入口
5. 只展开当前关心的 Authorization、OperationTracker 或 RunLifecycle 分支
6. `multi_agent/application/coordinator.py` 与 `live_fanout.py`
7. `bench/application/swebench.py`
8. `evaluation/application/scorecard.py`
9. `observability/application/usage.py`
10. `workbench/presentation/http.py` 中当前需要的 view renderer
