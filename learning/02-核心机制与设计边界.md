# 核心机制与设计边界

> 层级：[返回总入口](../README.md) → A3 主学习链 → 核心机制；顺序只以总入口为准。

本文只保留主演示和首轮技术追问所需机制。统一回答格式是：**问题 -> owner -> 机制 -> 失败场景
-> Evidence -> 不能夸大的边界**。

## 1. AgentLoop：模型决策外的确定性骨架

```text
prepare run
  -> prepare turn/context/tools
  -> call model and normalize response
  -> authorize and execute tools
  -> append observation/checkpoint
  -> continue or stop
```

`AgentLoop.run` 只拥有阶段顺序，不吞掉下游语义：

- `TurnPreparation` 组装完整 model request。
- `ToolExecutionPipeline` 拥有工具治理与副作用执行顺序。
- `OperationTracker` 拥有 operation identity、approval、stale 和 replay 语义。
- `RunLifecycle` 拥有 checkpoint、HITL、stop evidence 与终态。
- `AgentRunSession` 只保存当前 run 的进程内可变状态。

Runtime 通过 step、timeout、cost、重复 intent、连续失败和单 turn tool burst 停机，不依赖模型
“自觉结束”。不同参数但语义相同的 no-progress 仍没有通用判定，最终由硬预算停止并诊断。

## 2. Context：预算完整请求，而不是只数 messages

```text
hierarchical instructions
+ repository retrieval
+ activated Skill / recalled memory
+ working observations
+ tool schema
+ output reserve
  -> section budgets
  -> transaction-safe compaction
  -> ModelGateway
```

`ContextWindowManager.prepare` 对完整请求估算。压缩不能拆开 tool call 与对应 observation；摘要
保留 source hash，原始 trace 不删除。Provider 报告 overflow 后，只有压缩确实降低预算才重试一次。

当前 retrieval 是透明 lexical strategy，不声称 vector RAG 或 learned reranker。Skill/Memory 是
可选输入，不是理解主链所需的新入口。

## 3. Tool Governance：Prompt 不能授权副作用

```text
ToolCall
  -> task-aware visibility
  -> registry / schema validation
  -> permission policy
  -> command policy / workspace sandbox
  -> approval gate
  -> operation identity + stale/replay check
  -> ToolGateway.execute
  -> Observation + trace + ledger
```

| 门 | 防止的失败 |
| --- | --- |
| Visibility / schema | 暴露无关工具、未知工具、缺字段或错误类型 |
| Permission / command policy | 当前 mode 不允许的动作、网络或破坏性 shell |
| Workspace sandbox | 绝对路径、`..`、symlink 等逃逸 workspace |
| Approval | 未经人确认执行具体写副作用 |
| Operation tracker/ledger | resume 重复执行，或目标漂移后沿用旧批准 |

Local sandbox 是应用层路径边界；worktree 隔离 Git 状态；OCI 再增加 process、network 和 resource
boundary。它们都不等于 hostile multi-tenant security。

## 4. HITL：补信息与授权不能混为一谈

| 控制 | 回答的问题 | Durable state | 是否授权副作用 |
| --- | --- | --- | --- |
| Clarification | “缺少什么信息或选择？” | pending/responded/cancelled | 否 |
| Approval | “是否允许这个具体 operation？” | pending/approved/rejected/stale | 是 |
| Pause | “是否在安全点暂停？” | paused checkpoint | 否 |
| Cancel | “是否结束 continuation？” | cancelled terminal state | 否 |

同一 turn 同时出现 `ask_human` 与其他工具时，question 优先，其他副作用不执行；回答后模型必须
重新提出动作。状态必须先持久化再返回给调用者，否则进程在返回前崩溃会留下无法恢复的请求。

`forge demo` 使用确定性行为展示控制面，但两个 phase 都经 `Harness.run` 进入正式 Runtime；request、
checkpoint、resume 和工具执行没有旁路。它不证明模型推理质量。

## 5. Recovery：创建 continuation，不伪装恢复进程

```text
old checkpoint + human/approval decision
  -> continuation plan
  -> digest + resume chain
  -> new AgentLoop run
  -> operation replay/stale check
```

Resume 不恢复 Python stack、HTTP connection、模型隐藏状态或 KV Cache。可靠性来自：

- checkpoint 记录 completed/open state；
- operation key 来自 tool 与 canonical arguments 等稳定输入；
- ledger 记录副作用是否已经执行；
- target fingerprint 发现批准后的文件漂移；
- resume-chain artifact 连接旧 run 与新 run。

Pause/cancel/steer 在安全点被消费，不会强杀正在运行的命令，也不会假装回滚已完成副作用。
真实 continuation 用同一个 `forge resume` 同时保存人工决定并启动新 run：clarification 传
`--answer`，approval 传 `--decision approved|rejected`。

## 为什么这些抽象值得保留

| 抽象 | 不能直接内联的原因 | 删除后丢失 |
| --- | --- | --- |
| `AgentLoop` | 阶段顺序需要唯一 owner | stop/checkpoint/tool 顺序漂移 |
| `TurnPreparation` | 完整 context budget 是独立事务 | overflow 与 compaction 边界混入循环 |
| `ToolExecutionPipeline` | 模型意图和确定性执行必须隔开 | policy 可被 prompt 绕过 |
| `OperationTracker` | approval、identity、ledger、stale 共享同一不变量 | resume 重复或误用旧批准 |
| `RunLifecycle` | durable 状态必须先于返回/终止落盘 | 无法可靠 continuation |
| 外部 Port | 模型、文件、Git、进程是真实变化边界 | 测试替身与基础设施隔离消失 |

相反，只转发一次参数、复制字段或为了目录整齐存在的 Wrapper/Mapper 不能从“六边形架构”自动
获得保留理由。

## 追问层：知道入口和边界，不进主演示

### Memory

Working Memory 属于当前 run；Session Digest 服务 continuation；Long-term Memory 是跨 run、带
scope 和 evidence 审核的记录。长期记忆先是 candidate，再进入 active/superseded/retired/rejected。
它不是聊天 messages、用户画像、组织知识库或 KV Cache。

### Multi-Agent / Fanout

顺序 coordinator 通过显式 artifact handoff 协作；Live fanout 在 disposable worktree 中执行
validated DAG，检查 declared/actual scope，再确定性集成。scope overlap 串行化；scope escape、
patch conflict、worker failure 或 verifier mutation fail closed。当前不是 distributed swarm，也
不自动拆任务。

### MCP / Skills

MCP 是外部工具能力的 Adapter 边界，不能绕过 Tool Governance；Skill 是进入 Context 的版本化
指令/资源，不是另一套 Agent Runtime。两者仅在岗位追问时展开，不增加 Core Public Entry。

### 模型适配

`ModelCapabilities` 影响 context budget、parallel tool call 和 transport；`ToolCallNormalizer` 只修复
可确定解析的格式，不猜工具名或业务参数。适配目标是保持 Runtime contract，不为弱模型放宽安全。

## 闭卷检查

1. 从模型 ToolCall 画到 ToolGateway，解释每个门防止什么失败。
2. 用 stale approval 解释 operation key、fingerprint 和 ledger 的不同职责。
3. 区分 clarification、approval、pause、cancel；指出先持久化再返回的不变量。
4. 解释 resume 为什么是 continuation，不是 process snapshot。
5. 随机挑一个 Core 抽象，说明其独立语义；若只能说“更规范”，提出合并方案。

每题必须包含 owner、失败场景、Evidence 和边界；只背术语不得超过 1 分。
