# 核心机制与设计边界

本文不罗列全部功能，只保留面试中必须能讲出“问题 -> 机制 -> 证据 -> 边界”的七组设计。

## 1. AgentLoop：有边界的决策循环

```text
prepare run
  -> prepare turn/context/tools
  -> call model
  -> normalize response
  -> authorize and execute tools
  -> append observations/checkpoint
  -> continue or stop
```

`AgentLoop.run` 只编排阶段：

- `RunPreparation` 建立 task、Skill、初始上下文与 ambiguity gate。
- `TurnPreparation` 组装完整 model request 并执行 context window policy。
- `ToolExecutionPipeline` 拥有工具治理和副作用执行。
- `RunLifecycle` 拥有 checkpoint、HITL、终态和 stop evidence。
- `AgentRunSession` 只保存一次 run 的进程内可变数据。

停机不是依赖模型“自觉结束”。Runtime 同时限制 step、timeout、cost、完全相同的 tool intent、
连续失败和单 turn tool burst。不同参数但语义相同的 no-progress 目前没有通用判断，最后仍由
硬预算停止并进入 failure diagnosis。

## 2. Context：先分预算，再压缩

一次模型请求不只有聊天消息，还包括 system instruction、repository context、Skill、tool
schema 和输出预留。`ContextWindowManager.prepare` 对完整请求估算，不能只计算 message。

```text
hierarchical instructions
+ repository map / retrieval
+ activated Skill
+ recalled long-term memory
+ working memory / recent observations
+ tool schema
-> section budgets
-> transaction-safe compaction
-> model request
```

压缩不拆开 tool call 与对应 observation；摘要带 source hash，原始 trace 不删除。Provider
报告 overflow 后，只有压缩确实降低预算才重试一次，避免无效重试循环。

当前 retrieval 是透明 lexical strategy，不声称 learned reranker 或 vector RAG。

## 3. Memory：三层状态，不是一个 messages 列表

| 层 | 生命周期 | 用途 |
| --- | --- | --- |
| Working Memory | 当前 run | 近期事实、observation 和决策上下文 |
| Session Digest | continuation | 压缩后的已完成工作、开放问题与关键事实 |
| Long-term Memory | 跨 run、workspace/agent scoped | 经证据审核后可再次召回的稳定记录 |

长期记忆先进入 candidate，再由明确 evidence 提升为 active；旧事实可以 supersede、retire 或
reject。这样做是为了避免模型把猜测直接写成长期真相。

关键入口：`context/application/compaction.py`、`memory_service.py`、
`context/domain/memory.py`、`context/adapters/memory_json.py`。

当前没有用户画像、组织共享知识、多租户记忆或 KV Cache 恢复。

## 4. Tool Governance：Prompt 不能代替执行策略

```text
ToolCall
  -> ToolRouter visibility
  -> registry/schema validation
  -> permission hook
  -> command policy / workspace sandbox
  -> approval gate
  -> operation identity + stale check
  -> execute
  -> observation + trace + ledger
```

| 机制 | 防止什么 |
| --- | --- |
| ToolRouter | 与任务无关的工具扩大选择空间或暴露能力 |
| Schema validation | 非法 tool name、缺字段或错误参数类型 |
| PermissionPolicy | 当前 mode 不允许的 read/write/action |
| CommandPolicy | 网络、破坏性命令或不允许的 shell 形态 |
| WorkspaceSandbox | 绝对路径、`..` 或 symlink 逃逸 workspace |
| Approval | 未经人确认执行具体写副作用 |
| Operation ledger | resume 后重复执行，或目标变化后沿用旧批准 |

Local sandbox 是应用层边界，不是 OS isolation。Worktree 隔离 Git 状态；OCI 增加 process、
network 和 resource boundary，但仍不宣称 hostile multi-tenant security。

## 5. 两类 Human-in-the-loop

Clarification 和 Approval 必须分开：

| 类型 | 问题 | 持久状态 | 是否授权副作用 |
| --- | --- | --- | --- |
| Human Input | “缺少什么信息或选择？” | pending/responded/cancelled | 否 |
| Approval | “是否允许这个具体操作？” | pending/approved/rejected/stale | 是 |

模型在同一 turn 同时返回 `ask_human` 和其他工具时，human question 优先，其他副作用不执行；
回答后模型必须重新提出动作。这样避免“嘴上说等人，实际已经改文件”。

现场入口：

```bash
forge showcase hitl start
forge showcase approval start
```

Showcase 固定模型 tool call，但 request、checkpoint、store、resume 和工具执行都走正式 Runtime。
它证明控制面，不证明模型推理质量。

## 6. Recovery：恢复显式状态，不伪装恢复进程

```text
old run checkpoint
  -> continuation plan
  -> digest + answered human input + resume chain
  -> new run
```

Resume 不恢复 HTTP connection、Python stack、模型隐藏状态或 KV Cache。可靠性来自：

- checkpoint 明确记录 completed/open state。
- operation key 由 tool 与 canonical arguments 等稳定输入形成。
- ledger 记录副作用是否已执行。
- target fingerprint 检查批准后文件是否漂移。
- resume-chain artifact 连接旧 run 与新 run。

Pause/cancel/steer 也是协作式 safe point：Runtime 在 turn、模型返回和工具之间消费控制信号，
不会强杀正在执行的命令，也不会自动补偿更早的副作用。

## 7. Multi-Agent：Artifact、Ownership、Isolation

顺序 coordinator：

```text
Implementer artifact -> Reviewer artifact -> Verifier artifact -> bounded revision
```

角色不共享隐藏 memory；后续角色只读取显式 artifact。Reviewer/Verifier 的 tool view 受 Runtime
约束，不靠 prompt 自律。

Live fanout：

```text
validated DAG
  -> dependency + declared-scope batches
  -> disposable worktree workers
  -> actual touched-file check
  -> deterministic patch integration
  -> read-only finalizer + mutation gate
```

冲突策略不是“最后让一个 Agent 猜怎么合并”：声明 scope 重叠则串行；scope escape、patch
conflict、worker failure 或 verifier mutation 都 fail closed。Partial recovery 保存 plan/base
identity 和 patch hash，在新 integration workspace 重放 accepted patch，只重跑未完成 task。

当前不做自动任务拆解、distributed queue、peer swarm 或任意 conflict resolver。

## 模型适配边界

`ModelCapabilities` 影响 context budget、parallel tool call 和 transport 选择；
`ToolCallNormalizer` 只修复可以确定解析的格式，不猜工具名或业务参数。Native tool calling
不可用时可以使用受限单工具 JSON 文本协议。

模型适配的正确目标是保持 Runtime contract，不是为弱模型悄悄放宽安全或伪造成功。

## 闭卷检查

1. 不看资料画出 ToolCall 从模型到工具执行的八个门。
2. 用一个 stale approval 场景解释 fingerprint 和 ledger 分别解决什么。
3. 区分 clarification、approval、pause、cancel 四种人工控制。
4. 解释 resume 为什么是 continuation，而不是 process snapshot。
5. 说明 working memory、digest、long-term memory 的写入者、生命周期和权威性。
6. 比较顺序 coordinator 与 live fanout，并说出三种 fail-closed 条件。
7. 回答“Agent 一直空转怎么办”，不得提 Fanout DAG。

每题必须包含一个代码入口、一个失败场景和一个不能夸大的边界，才算掌握。
