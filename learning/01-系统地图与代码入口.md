# 系统地图与代码入口

目标不是记住全部文件，而是看到任意能力时，都能回答：入口在哪、谁编排、状态是什么、
副作用归谁、证据写到哪里。

## 一句话模型

```text
外部请求
  -> Harness / CLI / Benchmark Use Case
  -> AgentLoop 阶段编排
  -> Context / Model / Tool
  -> Safety / State / Evidence
  -> Patch / Evaluation / Diagnosis
```

模型提出意图；Harness 控制可见上下文、工具执行、状态恢复和结果验收。

## 地图一：运行入口

| 场景 | 外围入口 | 第一层 Use Case |
| --- | --- | --- |
| 嵌入式运行 | `agent_forge/harness.py::Harness.run` | `runtime/application/agent_loop.py::AgentLoop.run` |
| CLI 运行 | `cli/dispatch.py::main` | `cli/repository.py::run_repository_task` |
| 恢复任务 | `Harness.resume` / `cli/resume.py` | `operator_control.py::BuildContinuationPlan.execute` |
| 顺序多 Agent | `multi_agent/api.py` | `MultiAgentCoordinator.run` |
| 并发 Fanout | `multi_agent/api.py` | `LiveFanoutCoordinator.run` |
| SWE-bench | `bench/api.py::run_swebench` | `RunSwebench.execute` |
| Repeated Campaign | `bench/api.py::run_benchmark_campaign` | `RunBenchmarkCampaign.execute` |
| 本地工作台 | `workbench/presentation/http.py::run_ui` | Workbench query/command service |

第一遍只打开右列方法，不从 CLI parser 或 JSON adapter 开始。

## 地图二：依赖方向

```text
presentation / public API
          |
          v
      application  ------> ports <------ adapters
          |
          v
        domain
```

| 层 | 用一句话判断 | 例子 |
| --- | --- | --- |
| Domain | 不访问外部世界的状态和规则 | `TaskCheckpoint`、`OperationRecord`、failure taxonomy |
| Application | 编排一个 use case | `AgentLoop`、`ToolExecutionPipeline`、`RunLifecycle` |
| Port | Application 需要但不关心实现的契约 | Model、Tool、State、Event、Environment Protocol |
| Adapter | 对接文件、Git、HTTP、模型或进程 | JSON repository、OpenAI client、worktree、OCI |
| Presentation | 把外部输入翻译为 use case，并渲染结果 | CLI、HTTP Workbench、Markdown report |
| Wiring | 选择具体实现并组装对象 | `runtime/wiring.py::build_agent_loop` |

判断技巧：出现 `Path.write_text`、`subprocess` 或 HTTP，通常已到 Adapter；出现状态迁移但
没有 I/O，通常属于 Domain；同时调用多个 Port 完成目标，通常属于 Application。

## 地图三：数据流

```text
RunRequest
  -> RuntimeConfig + RuntimeDependencies
  -> AgentRunSession
  -> ContextAssemblyRequest / ContextBuildReport
  -> AgentResponse / ToolCall
  -> Observation
  -> TaskCheckpoint + TraceEvent
  -> RunResult / patch / scorecard
```

最值得先认的六组类型：

| 类型 | 解决的阅读问题 |
| --- | --- |
| `RunRequest` / `RunResult` | 外部到底传入和得到什么 |
| `RuntimeConfig` / `RuntimeDependencies` | 策略与依赖怎样进入 AgentLoop |
| `AgentRunSession` | 一次 run 的进程内可变状态在哪里 |
| `Message` / `ToolCall` / `Observation` | 模型与工具之间交换什么 |
| `TaskCheckpoint` | 中断后持久保存什么 |
| `TraceEvent` / scorecard models | 运行事实怎样投影为证据 |

不要把 `AgentRunSession` 当成聊天 session，也不要把 `TaskCheckpoint` 当成进程快照或 KV
Cache。

## 地图四：状态流

单 Agent task：

```text
created -> running
            |-> waiting_human -> running
            |-> waiting_approval -> running
            |-> paused -> resumed run
            |-> blocked / failed / cancelled / completed
```

副作用 operation：

```text
planned -> waiting_approval -> approved -> executed
                         |-> rejected
approved + target drift -> stale
```

Fanout：

```text
validated plan
  -> dependency batches
  -> isolated workers
  -> scope/result gates
  -> deterministic integration
  -> read-only finalizer
```

三个状态机彼此关联但不能混讲。Single Agent 空转由 step、timeout、cost、repeat 和 failure
budget 停止，不是靠 Fanout DAG 检测环。

## AgentLoop 最短阅读路径

折叠全部方法后，只按这个顺序展开：

1. `Harness.run`：外部输入怎样成为内部 request。
2. `build_agent_loop`：哪些 Port/Adapter 被装配。
3. `AgentLoop.run`：start、prepare、turn、tool、stop 的骨架。
4. `TurnPreparation.execute`：上下文和工具 schema 怎样进入模型调用。
5. `ToolExecutionPipeline.execute_calls`：工具如何被治理和执行。
6. `RunLifecycle.update/stop`：checkpoint、HITL 与终态怎样落盘。

每次只向下追一层。Adapter 的 JSON 序列化、renderer 和 accessor 在第一遍可以折叠。

## 三条关键链路

普通 run：

```text
Harness.run
  -> build_agent_loop
  -> AgentLoop.run
  -> TurnPreparation.execute
  -> ModelGateway.chat
  -> ToolExecutionPipeline.execute_calls
  -> RunLifecycle.update
```

HITL continuation：

```text
ask_human ToolCall
  -> RunLifecycle.request_human_input
  -> JsonHumanInputRepository.request
  -> waiting_human checkpoint
  -> forge respond
  -> BuildContinuationPlan.execute
  -> new AgentLoop run
```

Benchmark：

```text
run_swebench
  -> RunSwebench.execute
  -> clean checkout + LocalCaseExecutor.run
  -> AgentLoop
  -> patch / trace / usage
  -> official result parser
  -> scorecard / failure diagnosis / report
```

## PyCharm 低噪音读码法

不要用一次全局 Find Usages 代替调用链分析。每个动作回答的问题不同：

| 目的 | PyCharm 动作 | 用法 |
| --- | --- | --- |
| 看本文件全貌 | File Structure（`Command+F12`） | collapse all 后按名字定位 `run/execute`，不滚动找方法 |
| 跳到定义 | Go to Declaration（`Command+B`） | 看静态类型、数据结构和被调用方法 |
| 找具体实现 | Go to Implementation（`Option+Command+B`） | 从 Port/Protocol 跳到 JSON、模型或执行环境 Adapter |
| 看谁调用谁 | Call Hierarchy（`Control+Option+H`） | Caller 看上游入口，Callee 看下游依赖；优先于普通 usages |
| 查全部引用 | Find Usages（`Option+F7`） | 只在确认 API 影响面时使用，Scope 选 production |
| 快速扫少量引用 | Show Usages（`Option+Command+F7`） | 先看弹窗；结果确实复杂再固定到 Find 窗口 |
| 回到上一节点 | Navigate Back（`Command+[`） | 沿链路深挖后原路返回，不靠标签页猜位置 |
| 找任意动作 | 双击 `Shift` / `Shift+Command+A` | 忘记快捷键时搜索目标或动作名，不靠菜单逐层寻找 |

本机 NanoHarness 已配置五个 Scope：`00 Review Path`、`10 Production Code`、
`20 Runtime Control Plane`、`30 Evaluation Loop`、`90 Tests`。在 Project 窗口顶部、
Find Usages 的 Scope 或 Call Hierarchy 的 Scope 中选择它们。

多设备不需要手工重配。先用 PyCharm 打开 NanoHarness 一次，再在本学习仓库运行：

```bash
python3 scripts/install_pycharm_navigation.py /path/to/NanoHarness
python3 scripts/install_pycharm_navigation.py --check /path/to/NanoHarness
```

脚本只修改目标仓库中被 Git 忽略的 `.idea`：安装五个 Scope、标记 `tests`，并排除虚拟环境、
缓存、构建目录和运行 artifact；修改 `.iml` 前会保留一次本地备份。

推荐固定动作：

1. Project 窗口先选 `00 Review Path`，只看九个主链文件。
2. 在目标文件 collapse all，用 File Structure 直接打开主要入口。
3. 对调用语句用 `Command+B` 向下；遇到 Port 用 Go to Implementation 看 Adapter。
4. 对方法声明开 Call Hierarchy，切 Caller，并把 Scope 设为 `10 Production Code`。
5. 必须看全部引用时才用 Find Usages；Scope 仍选 production，关闭 import usages。
6. 若已经搜了全项目，在结果中启用 Group By -> Test/Production，把 Tests 整组折叠。
7. 主链说清后再切 `90 Tests`，只读一个最能证明行为边界的测试。

绿色测试结果不是“噪音垃圾”，只是第二遍证据。第一遍研究运行关系时隐藏，确认行为和边界时
再打开。Find Usages 结果中的某个分组也可以临时 Exclude，不会删除源码。

IDE 结果可用终端快速复核：`rg -n "主要入口" agent_forge --glob '*.py'` 列出显式入口；
`rg -n '\bAgentLoop\b' agent_forge --glob '*.py'` 只查生产包，不让测试命中混入结果。

## 闭卷检查

合上本文，用白纸完成以下任务：

1. 画出普通 run 的七个节点，并给每个节点写 owner 文件。
2. 解释 Domain、Application、Port、Adapter 的依赖方向，并举一个真实类。
3. 区分 `RunRequest`、`AgentRunSession`、`TaskCheckpoint`、`RunResult`。
4. 解释为什么 Single Agent no-progress 不能用 Fanout DAG 回答。
5. 在 IDE collapse all 后，五分钟内定位 HITL 是在哪里让 run 停下的。

任何一题只能说概念、指不出入口，最多记 1 分；能沿调用链定位并说明产物才记 3 分。
