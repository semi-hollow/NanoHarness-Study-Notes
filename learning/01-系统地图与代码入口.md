# 系统地图与代码入口

目标不是背文件树，而是看到任意重要 symbol 时，都能回答：它在流程哪里、谁调用、调用谁、
拥有哪条规则、产生什么 evidence、删除后会破坏什么。

先用主仓的 [Facade 目录](https://github.com/semi-hollow/NanoHarness/blob/master/docs/architecture/facade-catalog.md)
确认“项目能做哪些动作”；本文只负责从动作进入规范代码 owner，不重复 CLI 参数。

## 唯一黄金主链

```text
repository issue / base commit
  -> forge run（薄 CLI）
  -> RunRequest
  -> Harness.run（唯一 Runtime Public API）
  -> runtime wiring
  -> AgentLoop.run
  -> TurnPreparation -> ModelGateway
  -> ToolExecutionPipeline -> OperationTracker / ToolGateway
  -> RunLifecycle -> TaskCheckpoint
  -> RunResult + candidate artifacts
  -> RunManifest -> Run Story（Single-Run facts）
```

Local/Official 不由 `RunResult` 或 Run Story 自行生成；SWE-bench 外层另行执行本地验证和官方
Evaluator，再把类型化 case result 投影到 Scorecard。

第一遍只学习这条链。`forge demo` 是受引导的展示入口，`forge inspect` 是读取同一份 Run Story
的入口；它们不拥有 Runtime 装配、状态判断或 Evidence 语义。

## 六边形架构：两种箭头必须分开

六边形架构保留，因为模型、文件、Git、进程和 evaluator 都是真实外部边界。但分层本身不是
价值；每次跳转必须增加新语义。

编译期依赖方向：

```text
presentation / public API
          |
          v
      application  ------> ports <------ adapters
          |
          v
        domain
```

运行时调用方向：

```text
CLI -> Harness -> Application -> Port -> injected Adapter -> external system
```

Application 只依赖 Port 的类型，但运行时会通过注入对象调用 Adapter。不要因为运行时进入了
JSON repository，就误说 Application 在编译期依赖 JSON 实现。

| 层 | 存在理由 | 真实例子 |
| --- | --- | --- |
| Domain | 表达不依赖 I/O 的状态、不变量和转换 | `TaskCheckpoint`、`OperationRecord` |
| Application | 编排一个有业务语义的 use case | `AgentLoop`、`ToolExecutionPipeline`、`RunLifecycle` |
| Port | 隔离真实外部边界，或提供有价值的测试替身 | Model、State、Environment、Tool Protocol |
| Adapter | 连接文件、Git、HTTP、模型、容器或进程 | JSON store、model client、worktree、OCI |
| Presentation | 翻译输入并渲染结果 | CLI、Workbench、report |
| Wiring | 在唯一位置选择具体实现 | `runtime/wiring.py` |

若一个 Wrapper、Service、Port 或 DTO 既不表达状态/安全边界，也不隔离外部系统、不消除多个
调用方的重复，它不能仅凭“分层更规范”进入 Core。

## 12 文件首轮预算

`00 NanoHarness Review Path` 只保留下面 12 个文件。顺序就是阅读顺序，不从 parser、renderer
或 Adapter 序列化开始：

| # | Owner 文件 | 只回答一个问题 |
| --- | --- | --- |
| 1 | `agent_forge/harness.py` | Public request/result 与单次 run 边界是什么 |
| 2 | `agent_forge/runtime/wiring.py` | 哪些 Port/Adapter 被唯一装配 |
| 3 | `agent_forge/runtime/application/agent_loop.py` | 一次 run 的阶段怎样编排 |
| 4 | `agent_forge/runtime/application/session.py` | 进程内可变状态放在哪里 |
| 5 | `agent_forge/runtime/application/turn_preparation.py` | Context、Skill、工具 schema 怎样进入模型请求 |
| 6 | `agent_forge/runtime/application/tool_execution.py` | ToolCall 怎样被确定性治理并执行 |
| 7 | `agent_forge/runtime/application/operation_tracker.py` | 副作用怎样获得 identity、approval 与幂等记录 |
| 8 | `agent_forge/runtime/application/run_lifecycle.py` | checkpoint、HITL、stop 和终态由谁拥有 |
| 9 | `agent_forge/runtime/domain/task.py` | durable task state 的不变量是什么 |
| 10 | `agent_forge/runtime/domain/operation.py` | operation 状态机为何不能删 |
| 11 | `agent_forge/observability/domain/event.py` | 运行事实怎样成为 trace event |
| 12 | `agent_forge/observability/domain/run_story.py` | Artifact 血缘怎样投影为 canonical Run Story |

这不是整个项目只能有 12 个文件，而是面试首轮必读预算。Context、Memory、Multi/Fanout、MCP、
Skills、Campaign 和各 Adapter 在追问时从对应 Port 向外展开。

### Evaluation Follow-up：只展开五个 owner

| Owner | 面试只回答什么 |
| --- | --- |
| `bench/application/swebench.py::RunSwebench.execute` | case 执行、prediction、optional official、汇总的顺序 |
| `bench/domain/catalog.py::SMOKE_5_PROFILE` | 为什么选这组机制样本、为什么不能外推 Lite 总体 |
| `evaluation/domain/scorecard.py::aggregate_cases` | candidate/local/official 分母怎样聚合 |
| `evaluation/domain/ablation.py::compare_benchmark_scorecards` | matched identity 和单因素比较怎样拒绝 drift |
| `bench/domain/failure_taxonomy.py::classify_case_result` | 失败怎样指回下一次应修改的 owner |

JSON 读写、Markdown/HTML renderer、official report 兼容解析和临时目录清理属于 Adapter/Presentation；
知道输入输出即可，不进入首轮源码。

## 四组状态对象

```text
RunRequest
  -> RuntimeConfig + RuntimeDependencies
  -> AgentRunSession
  -> Message / ToolCall / Observation
  -> TaskCheckpoint + TraceEvent
  -> RunResult + artifacts
```

| 类型 | 生命周期 | 不可替代的语义 |
| --- | --- | --- |
| `RunRequest` | 调用开始前 | 外部请求与运行约束 |
| `AgentRunSession` | 当前 Python run | 进程内阶段状态，不是持久 session |
| `TaskCheckpoint` | 跨中断 | 可恢复的 durable continuation state |
| `RunResult` | 调用结束后 | Public API 返回结果，不替代 official evaluation |

不要把 `TaskCheckpoint` 说成 Python stack、HTTP connection 或 KV Cache 快照。

## 三条 Core 调用链

普通 run：

```text
forge run -> Harness.run -> build_agent_loop_from_request -> AgentLoop.run
  -> TurnPreparation.execute -> ModelGateway.chat
  -> ToolExecutionPipeline.execute_calls -> RunLifecycle.update/stop
```

Approval/recovery：

```text
ToolCall -> OperationTracker -> waiting_approval checkpoint
  -> forge resume <run> --decision approved|rejected
  -> persist decision -> continuation plan -> new AgentLoop run
  -> ledger replay/stale check -> execute at most once
```

Single-Run Evidence：

```text
TraceEvent + TaskCheckpoint + candidate artifacts
  -> RunManifest -> canonical Run Story

Benchmark case + candidate patch
  -> recorded Local Validation
  -> independent Official Evaluator
  -> typed case result / Scorecard
```

Multi/Fanout 是 Advanced coordinator：复用 Single-Agent worker Runtime，但不进入这三条 Core 链。

## Code Compass：看见任意 symbol 后怎么定位

Core 模块、类和关键方法的 docstring/导航注释应直接给出五件事：

```text
黄金主链位置
规范上游 caller
下一语义 owner
拥有的不变量或副作用
产出的状态 / artifact / evidence
```

“规范上游”只标生产主链调用者，不复制所有测试、兼容入口和间接 caller；完整影响面仍以 IDE
Call Hierarchy/Find Usages 为准。注释不应复述语句，而应解释“为什么不能直接删”。

### 随机 symbol 六问

每轮从 `10 Production Code` 随机点开一个 class 或 public method，限时两分钟：

1. 它属于 Domain、Application、Port、Adapter、Presentation 还是 Wiring？
2. 规范上游是谁？通过哪种状态/参数调用？
3. 下一语义 owner 或外部边界是谁？
4. 它拥有哪条规则、状态转换或副作用？
5. 哪个 trace、checkpoint、artifact 或测试能证明它的行为？
6. 删除/内联会失去什么？若答不出，是否应降级、合并或删除？

最后一问不能先假设“删了肯定不行”。没有独立语义的抽象，本来就应进入审计。

先生成机器辅助导航卡，再回到 IDE 验证静态关系与动态注入边：

```bash
forge inspect AgentLoop.run
```

Code Compass 的 AST caller/callee 不是完整运行时调用图；动态注入边以 Core owner 契约为准。

### IDE 最短动作

| 问题 | PyCharm 动作 | 使用边界 |
| --- | --- | --- |
| 本文件有哪些入口 | File Structure / collapse all | 先找 `run/execute/update`，不滚动猜 |
| 静态类型在哪里 | Go to Declaration | 看 Domain/Port 契约 |
| Port 的实现是谁 | Go to Implementation | 只在需要理解外部边界时展开 Adapter |
| 谁调用这个方法 | Call Hierarchy -> Caller | Scope 选 `10 Production Code` |
| 它又调用谁 | Call Hierarchy -> Callee | 每次只追到下一个语义 owner |
| 修改影响面多大 | Find Usages | 最后才查，并折叠 tests/imports |

## 安装低噪音 Scope

先用 PyCharm 打开 NanoHarness 一次，再在本仓库运行：

```bash
python3 scripts/install_pycharm_navigation.py /path/to/NanoHarness
python3 scripts/install_pycharm_navigation.py --check /path/to/NanoHarness
```

脚本只修改目标仓库中被 Git 忽略的 `.idea`，安装 `00 Review Path`、Production、Runtime、
Evaluation 和 Tests Scope，并排除虚拟环境、缓存、构建目录与 run artifacts。

固定顺序：先用 `00 Review Path` 走通黄金主链；再对一个关键方法打开 Caller；最后到 `90 Tests`
只读一个最能证明边界的测试。绿色测试不是主链说明，只是第二层行为证据。

## Artifact 定位入口

Symbol 的静态位置由本章负责；artifact 的 producer、consumer、来源和 claim boundary 由
[Benchmark 与证据闭环](03-Benchmark与证据闭环.md)负责。不要靠文件名猜语义，使用
`forge inspect <run-or-artifact>` 进入统一 Run Story。

## 闭卷检查

1. 画出黄金主链，并给每个节点写一个 owner。
2. 用同一个 Tool Gateway 解释编译期依赖和运行时调用为何箭头不同。
3. 区分 `RunRequest`、`AgentRunSession`、`TaskCheckpoint`、`RunResult`。
4. 随机选择一个 Core symbol，完成“层级、上游、下游、不变量、Evidence、删除影响”六问。
5. 解释为什么保留六边形边界不等于保留所有 Port/Wrapper。

只能说概念、不能从 caller 定位 owner，最多 1 分；能说明删除影响并指向证据才记 3 分。
