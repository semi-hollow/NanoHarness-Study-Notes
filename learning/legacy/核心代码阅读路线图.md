# NanoHarness / Agent Forge 核心代码地图

> 目标：用最少而精的代码文件，理解项目设计骨架、依赖关系和三条主链路；面试时能清楚说明它为什么不是 toy project。

## 0. 一句话架构

Agent Forge 的核心不是“堆很多 agent”，而是一个可治理的 agent runtime：

```text
CLI / Benchmark 入口
  -> RuntimeConfig + ToolRegistry + LLM Gateway
  -> AgentLoop 单 agent 内核
  -> Trace / Usage / Artifacts 证据层
  -> MultiAgentCoordinator 外层工作流
  -> SWE-bench Compare / Evaluation 评估层
```

面试时可以这样说：

> 这个项目的核心是 coding-agent 控制平面。`AgentLoop` 是 runtime kernel，`ToolRegistry/ToolRouter/Sandbox` 是 action boundary，`Trace/Usage` 是 observability，`MultiAgentCoordinator` 是 workflow layer，`SWE-bench compare` 是 evaluation layer。

## 1. 最小 7 文件核心版

如果你只想用最短时间抓住主线，先读这 7 个文件。

| 顺序 | 文件 | 一句话作用 | 为什么核心 |
| --- | --- | --- | --- |

| 3 | `agent_forge/runtime/agent_loop.py` | 单 agent 控制循环 | 项目的 runtime kernel |

## 3. 模块依赖关系

### 3.1 总依赖图

```text
forge_cli.py
  -> runtime.wiring.build_registry()
      -> safety.sandbox.WorkspaceSandbox
      -> tools.registry.ToolRegistry
      -> concrete tools: read/list/grep/apply_patch/run_command/...
  -> runtime.wiring.build_llm()
      -> models.gateway.ModelGateway
      -> runtime.llm_client.OpenAICompatibleLLMClient
  -> runtime.config.RuntimeConfig
  -> observability.trace.TraceRecorder
  -> AgentLoop.run()                         # single path
  -> MultiAgentCoordinator.run()             # multi path

swebench.py
  -> workspace manager / case loader
  -> _run_case(single)
      -> AgentLoop.run()
  -> _run_case(multi)
      -> MultiAgentCoordinator.run()
  -> _run_compare_case(compare)
      -> extract_run_metrics()
      -> compare_runs()
      -> write_evaluation_artifacts()
```

### 3.2 Runtime 内核依赖

```text
AgentLoop
  receives:
    RuntimeConfig
    TraceRecorder
    ToolRegistry
    ModelGateway-compatible LLM

  owns/uses:
    ToolRouter
    StepController
    Guardrails
    ContextBuilder / repo_map / memory
    HookManager / permission checks
    EvidenceLedger

  emits:
    trace events
    observations
    final_answer
    task state checkpoints
```

重点：`AgentLoop` 不自己创建全局依赖，而是由 CLI/bench 传入 config、trace、registry、llm。这个设计让 single run、SWE-bench run、multi-agent role run 共享同一套 runtime 语义。

### 3.3 Multi-agent 依赖

```text
MultiAgentCoordinator
  receives:
    task
    AgentProfile
    RuntimeConfig
    TraceRecorder
    base ToolRegistry
    LLM
    run_dir

  uses:
    RoleSpec               # role 能做什么
    ArtifactStore          # role 输出怎么交接
    AgentLoop              # 每个 role 仍然走同一个 runtime

  emits:
    multi_agent/artifacts/*.md
    multi_agent/artifact_index.json
    multi_agent/multi_agent_summary.json
    multi_agent/multi_agent_report.md
```

重点：Coordinator 是 workflow layer，不是第二套 agent runtime。

## 4. 三条主链路

### 4.1 Single-agent 链路

```text
forge run "task" --agent-mode single
  -> forge_cli.run_repository_task()
  -> build_registry(workspace)
  -> build_llm(llm_config)
  -> RuntimeConfig(...)
  -> AgentLoop(config, trace, registry, llm).run(task)
  -> trace.write()
  -> write_usage_artifacts(trace_path)
  -> final_answer.txt
```

你要看懂的是：模型不是直接改文件，而是每轮提出 tool call，runtime 通过 registry/router/sandbox/hook 执行，再把 Observation 喂回模型。

### 4.2 Multi-agent 链路

```text
forge run "task" --agent-mode multi --profile coding_fix
  -> get_profile("coding_fix")
  -> MultiAgentCoordinator.run()
      -> AgentLoop.run(role="Implementer")
      -> ArtifactStore.write_role_artifact()
      -> AgentLoop.run(role="Reviewer")
      -> parse first-line decision marker
      -> AgentLoop.run(role="Verifier")
      -> parse first-line decision marker
      -> if NEEDS_REVISION and budget remains:
           run Implementer again with feedback artifacts
      -> write summary/report
```

这里的核心不是“更多模型调用”，而是把实现、审查、验证拆成有权限边界的 roles，并把交接内容落盘。

### 4.3 Compare 链路

```text
forge bench swebench --agent-mode compare
  -> run_swebench(agent_mode="compare")
  -> _run_compare_case(case)
      -> _run_case(agent_mode="single", output_dir=case_root/single)
      -> _run_case(agent_mode="multi", output_dir=case_root/multi)
      -> extract_run_metrics(single artifacts)
      -> extract_run_metrics(multi artifacts)
      -> compare_runs(task_id, single_metrics, multi_metrics)
      -> write_evaluation_artifacts(case_root)
```

公平性的关键：single 和 multi 不共享一个可变 checkout，避免第一个 run 的 patch 污染第二个 run。
