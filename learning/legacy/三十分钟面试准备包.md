# 30 分钟面试压缩包

目标：用最短时间把 Agent Forge 讲成一个 3 年+ AI Agent 工程项目，而不是把所有文档都读完。

## 0. 先记住一句话

> Agent Forge 是面向真实工程任务的 Agent Harness 控制平面。它用 `AgentLoop` 管理上下文、工具、权限、HITL、恢复、trace 和 usage；用顺序 role coordinator 做 artifact review，用 live fanout 在独立 worktree 中执行显式 DAG；再用 evaluation evidence 解释质量、成本和失败模式。

不要把它讲成：

- 一个聊天壳子。
- 一个多 agent 炫技 demo。
- 一个已经有官方 resolved rate 的 benchmark 系统。

## 3. 核心代码依赖关系

第一遍先看 12 个文件：

```text
forge_cli.py
  -> runtime/wiring.py
  -> runtime/agent_loop.py
  -> runtime/human_input.py
  -> runtime/approval.py
  -> multi_agent/profiles.py
  -> multi_agent/coordinator.py
  -> multi_agent/fanout.py
  -> multi_agent/live_fanout.py
  -> multi_agent/artifacts.py
  -> runtime/git_workspace.py
  -> bench/swebench.py
```

面试解释顺序：

1. `forge_cli.py`：用户入口，决定 single / multi / compare。
2. `wiring.py`：组装 LLM、tools、sandbox，不把依赖散落到各处。
3. `agent_loop.py`：真正的 agent runtime 内核。
4. `human_input.py` / `approval.py`：信息输入和写授权两个状态机。
5. `profiles.py` / `coordinator.py`：顺序运行 roles，控制 revision loop。
6. `fanout.py` / `live_fanout.py`：DAG batching 到真实隔离 worker、merge 和 resume。
7. `artifacts.py`：把顺序角色通信落盘，避免隐藏上下文污染。
8. `git_workspace.py`：统一 tracked/untracked candidate patch 事实。
9. `swebench.py`：把项目接到真实 issue / benchmark 任务形态。

## 5. 当前项目已经解决什么

- 有真实 `AgentLoop`，不是一次 prompt 调用。
- 有 ToolRouter / ToolRegistry / sandbox / command policy。
- 有 approval queue / operation ledger / resume chain，能解释 HITL 和 partial recovery。
- 有 durable `ask_human` stop/respond/resume，不再用模拟回答。
- 有 trace / usage / report，能解释一次 run 为什么成功或失败。
- 有 coordinator-driven multi-agent，不是 agents 自由聊天。
- 有真实 AgentLoop live fanout、worktree 隔离、scope gate、deterministic merge 和 selective recovery。
- 有 artifact handoff，让 reviewer/verifier 基于显式证据工作。
- 有 mini-cases，用小型 evidence scorecard 讨论非 coding Agent 场景。
- 有 SWE-bench 任务形态和 compare report，能做质量、成本、失败模式对比。
- 有 UI evidence view，能降低面试官理解成本。

## 6. 还要诚实承认什么

- 本机当前可能缺 Docker / official `swebench` harness，不能随口说 official resolved rate。
- `multi` 是顺序 role coordinator；`fanout` 是本地真并发 worker，但不是分布式 swarm。
- fanout 暂不跨 ephemeral worktree 复用逐操作手工写审批；该组合会 fail-fast。
- UI 是本地 workbench，不是完整 SaaS/IDE 产品。
- 当前项目更适合作为 AI Agent runtime/control-plane 作品，而不是生产级 agent platform。

## 7. 最容易被问的 5 个问题

### Q1: 为什么不是 toy project？

因为它不只做 prompt demo，而是把 coding agent 的 runtime 问题拆成 context、tool、policy、trace、usage、eval。toy project 通常只能展示一个成功答案，Agent Forge 能解释失败归因和成本。

### Q2: 为什么需要 multi-agent？

不是为了“更多 agent 更强”，而是为了把实现、审查、验证拆成有边界的角色。代价是更多 token 和延迟，所以必须通过 compare report 做 case-by-case tradeoff。

### Q3: Agent 怎么防止乱执行命令？

模型只提出 tool call。真正执行前要经过 ToolRouter、ToolRegistry、PermissionPolicy、WorkspaceSandbox 和 CommandPolicy。安全边界在 runtime，不靠 prompt 自觉。

### Q4: patch_generated 和 resolved 有什么区别？

`patch_generated` 只说明生成了候选 diff；`resolved` 必须由官方或 focused validation 证明。项目故意把这两个概念分开，避免夸大。

### Q5: 如果 agent 失败，怎么 debug？

先看 failure taxonomy：provider/config、context、tool/runtime、permission/sandbox、validation、official eval。不同失败改不同层，不是一律改 prompt。

### Q6: 你的 multi-agent 是否真的并发？

两个答案分开讲：`MultiAgentCoordinator` 是顺序 Implementer/Reviewer/Verifier；
`LiveFanoutCoordinator` 对显式 DAG 中 scope 不冲突的 task 创建独立 worktree、
fresh LLM 和真实 AgentLoop 并发执行。它不是自动 swarm。

### Q7: 中断恢复怎么证明不是口号？

普通 run 用 checkpoint、resume chain 和 operation ledger；fanout 每批原子写
checkpoint，恢复时校验 plan/base/patch SHA，只重跑 incomplete worker。硬中断
不恢复隐藏模型状态，正在执行的 batch 可能重跑。

### Q8: HITL 为什么有两个 store？

`HumanInputStore` 回答“继续需要什么信息”；`ApprovalStore` 回答“是否授权具体
副作用”，后者要绑定 fingerprint 和 stale detection。混成一个布尔值会错误
复用授权。`ask_human` 还是同轮副作用 barrier；同一响应里的写工具必须
等回答后由模型重新提出。
