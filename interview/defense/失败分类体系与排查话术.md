# Agent 失败分类体系与排查话术

目标：回答“Agent 失败了，你怎么定位和迭代？”

## 1. 一句话

> 不成熟的系统只说 pass/fail；成熟的 Agent harness 要把失败归因到 provider、context、tool/runtime、safety、validation、evaluation environment（评测环境）等层，并给出下一步动作。

## 2. 失败分类表

| 分类 | 典型信号 | 优先看哪里 | 下一步 |
| --- | --- | --- | --- |
| Provider / 配置 | API key 缺失、timeout、invalid response | `trace.json`, `usage_report.md` | 修 provider 配置或 fallback，不改 Agent prompt。 |
| 上下文检索（Context Retrieval） | 找错文件、没有相关代码证据 | `context_assembly`, selected files | 调整 file ranking / repo map / context budget。 |
| 工具协议（Tool Schema） | missing args、invalid JSON、unknown tool | `tool_call`, `tool_observation` | 改 tool schema（工具协议）、parser、修复提示（repair prompt）。 |
| 权限与 Sandbox | denied、external path、sensitive path | `permission_check`, hooks | 不绕过策略；缩小任务或申请审批。 |
| 命令与验证 | command exit_code != 0 | `run_command` observation | 看 stderr/stdout，做最小修复或标记 blocked。 |
| Patch 锚点不匹配 | old text not found / ambiguous | `apply_patch` observation | 重新读文件，选择唯一 anchor。 |
| 循环坍缩（Loop Collapse） | repeated tool call（工具调用） / max_steps | `recovery_decision` | 让 planner 改策略，减少工具暴露或提高 context。 |
| Multi-Agent 审查 | NEEDS_REVISION / BLOCKED | `multi_agent_report.md` | 看 Reviewer/Verifier artifact（产物），回到 Implementer。 |
| Human input state | waiting_human / cancelled / stale answer | request JSON + task checkpoint | 检查 thread/question/choices，不模拟回答或复活 terminal request。 |
| Fanout plan/scope | invalid plan / scope_violation / dynamic_conflict | `fanout_plan.json`, worker result | 修 DAG/ownership，不扩大 worker 权限。 |
| Fanout merge/recovery | merge_conflict / digest mismatch / missing patch | checkpoint + patch + base commit | 在 fresh workspace 校验 artifact，失败 task 重跑。 |
| Finalizer | NEEDS_REVISION / BLOCKED / verifier mutation | finalizer trace/usage/manifest | 区分候选 patch、验证不足和 verifier 副作用。 |
| 官方评测（Official Eval） | official_eval_failed / unavailable | `report.md`, SWE-bench output | 区分 patch 质量失败和本地 Docker 环境缺失。 |

## 3. 项目里的证据来源

| 证据 | 用途 |
| --- | --- |
| `trace.json` | 最细粒度执行链路。 |
| `usage.json` / `usage_report.md` | token、cost、latency、context、tool efficiency（工具效率）。 |
| `multi_agent_report.md` | 角色时间线（role timeline）、修订轮次（revision rounds）、artifacts（产物）。 |
| `comparison.json` | single/multi 扁平指标。 |
| `evaluation_report.md` | 面试可读 evidence card。 |
| `report.md` | SWE-bench run 汇总和 失败诊断（failure diagnosis）。 |
| `fanout_checkpoint.json` | plan/base identity、accepted tasks 和 patch hash。 |
| `fanout_report.md` | batch、scope、merge、resume、worker/finalizer cost。 |
| `human_input/<id>.json` | pending/responded/cancelled 问题状态。 |

## 4. 面试讲法

> 我不会只看最终答案。Agent 失败后，我先看 失败分类（failure taxonomy）：如果是 provider/config，就不该改 prompt；如果是 context，就改检索和上下文预算；如果是 tool schema（工具协议），就改工具协议；如果是 permission（权限）/sandbox，就说明 runtime 正确阻止了风险；如果是 official eval failure，才说明 patch 逻辑本身需要改。这样 badcase 能回流到正确层，而不是所有问题都靠 prompt 重试。

## 5. 和 evaluation report 的关系

`evaluation_report.md` 里的 `Failure Lens` 是这份 taxonomy 的压缩版，面试时优先打开它。
这份文档用于深挖时展开说明每类失败如何定位。

## 6. 真实开发案例入口

项目内持续维护的完整案例是：

`docs/evaluation/failure-driven-improvements.md`

本轮新增案例包括：synthetic ask-human、terminal/stale request、fanout 未接真实
worker、scope 归一化、untracked patch 丢失、runtime artifact 自污染、
checkpoint-only recovery、patch tamper、finalizer 隔离/成本遗漏、worker human
response 跨 run 丢失和 verifier PASS claim boundary。

回答“开发中最难的问题”时，不要背抽象分类。选一个案例按以下顺序讲：

1. 具体可复现现象。
2. 先看了哪些 trace/artifact，排除了什么。
3. 根因落在哪个状态机或数据契约。
4. 为什么选择窄修复而不是堆 prompt。
5. 用什么行为测试或真实 run 证明。
6. 修复后仍保留什么边界。
