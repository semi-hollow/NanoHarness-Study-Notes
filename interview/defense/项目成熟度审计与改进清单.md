# 项目成熟度审计

这份文档用于自查：哪些东西能支撑 3 年+ AI Agent 工程面试，哪些东西还容易被看成 toy。

## 已经有生产工程味道的部分

### 1. 使用真实 benchmark 闭环

项目不再把 calculator 或 webhook 这类自造例子当作核心证明，而是以 SWE-bench reference case 作为展示入口。

面试讲法：

> 我用 SWE-bench 的任务形态做闭环：真实 GitHub issue、base commit checkout、agent 修改、patch 输出、可选官方 harness 评测。这样比自造 demo 更接近 coding agent 的真实问题。

### 2. Runtime control plane 明确

核心不是“提示词很长”，而是 AgentLoop 管住：

- 上下文选择和压缩；
- LLM 调用；
- tool routing；
- permission（权限） / command policy；
- sandbox；
- 最大步数（max steps） / budget；
- trace / usage。

面试讲法：

> 复杂 Agent 不能把所有控制逻辑都塞进 prompt。prompt 负责表达任务和策略，runtime 负责权限、预算、状态、回放和失败恢复。

### 3. Multi-agent 是 由 Coordinator 驱动（coordinator-driven）

多 Agent 没有做去中心化聊天，也没有做 Raft / quorum（法定票数）/ swarm（群体式协作）。它复用 AgentLoop，由 MultiAgentCoordinator 顺序调用 Implementer、Reviewer、Verifier，并通过 artifact（产物）交接。

面试讲法：

> 我没有为了炫技做自由通信的 multi-agent。coding 场景里更重要的是责任边界：实现、审查、验证分别产出 artifact（产物），review/verifier 只用显式证据触发修订。

### 4. 失败可归因

失败不只显示 failed，而是分成 provider/config、context miss、工具失败（tool failure）、策略拦截（policy block）、runtime budget、official eval unavailable 等方向。

面试讲法：

> Agent 评测 不能只看最终成功率。执行轨迹（trajectory）、usage、工具失败（tool failure）、策略决策（policy decision） 才能告诉你下一轮该优化哪里。

### 5. 证据可视化

`forge ui` 的 Interview Evidence / Result Summary / Usage Dashboard / Trace Timeline 把原始 artifacts（产物）转成展示材料。

面试讲法：

> 我把可观测性做成面试和调试都能用的 evidence dashboard。因为真正落地时，团队需要快速解释一次 agent run 为什么成功或失败。

## 仍然容易被质疑的边界

### 1. 没有宣称官方 resolved rate

当前项目可以跑 SWE-bench 任务形态，也能生成 predictions。只有在安装 SWE-bench harness + Docker 并跑 `--evaluate` 后，才能说官方结果。

回答方式：

> 这是我保留的工程边界。我不会把 patch generated 当作 resolved。项目当前重点是 harness 设计和 证据闭环（evidence loop），resolved rate 需要更长时间跑官方评测。

### 2. Multi-agent 是顺序 coordinator，不是真并行

回答方式：

> 第一版故意不做并行。因为 coding agent 的主要风险不是 CPU 并发，而是上下文污染、重复执行、责任不清。先把 artifact（产物） handoff（显式产物交接） 和 revision loop 做清楚，再考虑并行工具调用或并行 reviewer。

### 3. 不是完整 IDE 产品

回答方式：

> 我没有把精力放在 TUI/IDE 插件，而是把能力放在 runtime、工具治理（tool governance）、trace、eval。面试项目应该先证明 agent engineering 核心，而不是做一个外壳很重的产品。

### 4. 真实使用能力还需要持续 dogfood

回答方式：

> 我会继续用它处理小型 repo 任务，用 trace/usage/badcase 回流优化 tool schema（工具协议）、context strategy 和 budget policy。这是从 demo 走向真实 agent 的路径。

## 当前最该展示的三句话

1. Agent Forge 是一个面向 SWE-bench 的 CodingAgent control plane，不是玩具聊天机器人（toy chatbot）。
2. 它把复杂度从 prompt 移到 runtime：上下文、工具、权限、预算、状态、trace、eval 都有明确模块。
3. 它的 multi-agent 不是自由聊天，而是 由 Coordinator 驱动（coordinator-driven） artifact（产物） workflow（产物工作流），能回答为什么需要 reviewer/verifier 以及代价是什么。
