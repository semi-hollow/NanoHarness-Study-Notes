# Agent 面经精练版

> 层级：[返回总入口](../../README.md) → B2 外部输入 → 只作题库；不提供项目答案。

> 基于 Clowder AI 项目（https://github.com/zts212653/clowder-ai）相关面经进行去重、压缩与分类整理。仅保留问题，不提供答案。

## 一、Memory 与上下文管理

### 1. 记忆分层与边界

- Memory 为什么要分层？各层分别解决什么问题？
- Shared State、长期记忆、知识库、任务状态、协作记录之间有什么区别？
- Session Memory、项目知识、用户画像、组织知识如何划分？
- Session、项目、用户、租户之间如何隔离，避免记忆污染？

### 2. 记忆权威与生命周期

- 什么内容有资格成为长期记忆或架构决策？
- 聊天消息如何从低权威信息晋级为正式知识？
- 记忆如何验证、合并、更新和退役？
- 多个 Agent 写入冲突记忆时如何裁决？
- 原始消息、摘要和正式决策，哪个才是真相源？

### 3. 检索与召回

- Grep、BM25、向量检索、混合检索和知识图谱如何取舍？
- 为什么不是所有知识都做 Embedding？
- 短期记忆为什么可以全文加载，长期记忆为什么必须按需召回？
- 如何支持中英文、同义词和跨项目检索？
- 外部知识库如何接入？

### 4. 压缩与恢复

- 长任务中发生上下文压缩时，如何避免丢失关键细节？
- 摘要能否替代原始消息和证据？
- Agent 很久没参与讨论，重新被 @ 时如何补齐中间信息？
- 一个任务持续多天时，如何保持目标、决策和进度一致？
- 单一 Session 中切换多个主题时，如何防止上下文串扰？

## 二、Multi-agent 路由与协作

### 1. 首 Agent 路由

- 用户发来任务后，第一个接球的 Agent 如何确定？
- 没有明确 @ Agent 时如何路由？
- 用户同时表达任务、闲聊和补充信息时如何判断主要意图？
- 什么条件下需要拉入新的 Agent？
- 路由由模型判断、规则判断，还是由人指定？

### 2. 协作模式

- 单 Agent、主子 Agent、Agent Team、Shared State 模式有什么区别？
- 多 Agent 与单 Agent 的交互协议分别有哪些坑？
- Agent 之间是否共享完整上下文？
- 如何做到每个 Agent 只看到相关信息，同时又能追溯全局？
- A2A 为什么自建，而不是直接使用现成协议？

### 3. 分工与交接

- 多个 Agent 如何拆分任务和职责？
- 如何固定接口契约，避免重复实现和联调失败？
- Agent 交接时需要传递哪些信息？
- 子 Agent 的结果如何验证，避免幻觉向上传播？
- 多人和多 Agent 同时开发时如何处理所有权与代码冲突？

### 4. 终止与防失控

- 两个 Agent 一直互相对话时如何终止？
- 终止条件应依据轮数、任务状态、工具进展还是外部事件？
- 如何防止循环路由、乒乓式 A2A 和重复调用？
- 没有强状态机时，如何保证任务流转不偏移？
- Agent 异常退出、网络中断或额度耗尽后，任务如何被重新接管？

## 三、Agent Harness 与模型适配

### 1. Harness 定位

- 为什么使用现成 Coding Harness，而不是直接调用模型 API？
- 模型和 Harness 如何解耦？
- Harness 的核心职责是什么？
- Agent CLI、模型和上层协作平台分别负责什么？
- Harness 未来会向什么方向演进？

### 2. 不同模型适配

- Opus、Gemini、Codex、国产模型分别适合什么任务？
- 弱模型 Tool Calling 不稳定时如何治理？
- 切换模型后，Tool Schema、Prompt、重试和验证机制需要如何变化？
- 如何评价不同模型的推理能力、工具能力、风格和成本？
- 不同 Thinking Level 如何选择？

### 3. 权限与工具链

- Subagent → Tool Call → Bash 的完整链路有哪些盲点？
- 模型权限、工具权限和系统权限如何分层？
- 参数、权限、Schema 和前置条件如何提前拦截错误请求？
- 如何减少无意义 Agent Loop 和 Token 消耗？
- 外部系统的账号、凭证和授权如何管理？

## 四、Skill、自进化与事故驱动治理

### 1. Skill 机制

- Skill 是什么，如何被发现、加载和复用？
- Skill 如何从真实任务和事故中沉淀出来？
- 如何证明原来十步的任务变成五步后没有漏步骤？
- ToC 的开放 Skill 与 ToB 的规范化 Skill 如何平衡？
- “能力市场”中的能力如何安装、治理和升级？

### 2. 自进化

- Agent 自进化具体指什么？
- 经验应该沉淀到 Memory、Skill、规则、代码检查器，还是模型训练？
- 什么经验只适合记录，什么经验必须固化为 Harness？
- Agent 如何开发自己的 Harness？
- 如何避免 Agent 修改自身系统时发生漂移或失控？

### 3. 事故驱动护栏

- 什么是事故驱动护栏？
- 如何从一次失败抽象出通用规则？
- Prompt 软约束、代码硬约束和 Eval 如何组合？
- 如何防止 Agent 过度设计、过度实现或用“后续优化”逃避闭环？
- 愿景驱动开发如何转化为可执行约束？

## 五、Eval、Benchmark 与效果验证

### 1. 系统级 Eval

- Memory、A2A、Tracing、Skill、自进化分别如何评测？
- Harness 修改后如何证明系统真的变好？
- 如何保证新功能上线后旧功能不漂移？
- 为什么做或不做每日 Eval 曲线？
- 离线 Benchmark 与真实线上任务如何结合？

### 2. 模型比较

- 模型版本升级后如何做回归评测？
- Gemini、Opus、Codex、国产模型如何比较？
- Thinking Level 的收益与成本如何评估？
- 如何计算模型的质量、延迟、成本和成功率 ROI？
- 哪些复杂度应该隐藏，不暴露给普通用户？

### 3. 真实环境评测

- 如何准备带真实工具、权限和失败注入的虚拟环境？
- Eval 是否需要真实代码仓、外部服务和 Sandbox？
- 如何验证 Tool Calling、权限控制和异常恢复？
- 如何评估长程任务和 Agent RL？
- 如何判断 Memory 或 Skill 被真正召回并改善了结果？

### 4. 组织效果

- 如何衡量组织知识沉淀是否产生价值？
- 应观察开发速度、返工率、合入率、缺陷率还是人工检查点？
- 如何证明知识库不是单纯变大，而是提高了任务成功率？
- 如何评估 Agent 在千人团队中的真实生产力？

## 最终压缩为 8 个核心主题

1. **Memory**：真相、权威、隔离、检索、压缩与恢复。
2. **Multi-agent**：路由、分工、交接、终止和防失控。
3. **Harness**：模型解耦、工具权限、弱模型适配和运行时约束。
4. **自进化**：Memory、Skill、硬护栏与事故驱动改进。
5. **Eval**：系统评测、模型 ROI、真实环境和组织效果。
6. **工程交付**：Worktree、Sandbox、大仓、多仓、测试与 CI/CD。
7. **产品架构**：分层设计、竞品差异、云端化和 Agentic Work OS。
8. **企业落地**：组织知识、平台治理、安全审计和规模化推广。
