# 技术面试防守材料阅读入口

这组文档不是普通项目文档，而是为了让你用最短时间把 Agent Forge 讲成一个 3 年+ AI Agent 工程项目。

开源给面试官看时，先让对方读仓库根目录的 `README.md`。如果对方追问“哪些能力是真的，哪些只是 demo”，打开 `docs/CAPABILITY_REALITY_MATRIX.md`。

## 分类

### `learn/`: 连续学习路径

只放你真正需要顺序学习的材料。

- `../../CAPABILITY_REALITY_MATRIX.md`
  - 能力真实性矩阵：哪些是真 runtime，哪些是 lightweight primitive，哪些不能夸大。
  - 面试官质疑“是不是包装 demo”时优先打开。

- `learn/三十分钟面试准备包.md`
  - 先读这份。
  - 30 分钟学习顺序、文档依赖关系、核心 7 文件、UI 展示顺序都在这里。

- `learn/核心代码阅读路线图.md`
  - 最小 7 文件核心。
  - single / multi / compare 三条调用链。
  - 第一遍只读这个，不要一上来读所有文件。

- `learn/最近新增Agent能力代码导览.md`
  - Durable HITL、resume、operation ledger、真实 AgentLoop fanout、scope/merge gate、Git patch 证据和 finalizer 的新增能力地图。
  - 面试官追问“中断恢复、人工审批、并发任务、通用 Agent 应用评估”时先打开这份。

- `learn/多Agent机制学习指南.md`
  - 怎么跑 multi-agent 和 compare。
  - 怎么看 `comparison.json` / `evaluation_report.md`。

- `learn/核心代码依赖关系学习图谱.md`
  - 比 `核心代码阅读路线图.md` 更细，适合第二轮看。
  - 不建议第一遍打开。

### `demo/`: 展示路线和证据

只放面试现场要打开或引用的材料。

- `demo/五分钟面试演示脚本.md`
  - 5 分钟怎么展示。
  - 现场打开哪些 UI/artifacts（产物）。
  - 哪些话不能夸大。

- `demo/evidence/演示证据目录说明.md`
  - 当前可引用的真实运行证据。
  - 哪些指标能讲，哪些不能夸大。
  - 失败快照应该怎么转成工程解释。

### `defense/`: 面试攻防单点补充

这些不是学习主线。只有被问到对应问题时再打开。

- `defense/项目成熟度审计与改进清单.md`
  - 哪些地方不像 toy。
  - 哪些边界需要诚实承认。
  - 面试官质疑成熟度时怎么答。

- `defense/AI智能体项目面试问答.md`
  - 高频 12 问的短答案。
  - 面试前最后复习这份。

- `defense/Agent安全边界与权限防守说明.md`
  - 用于回答“Agent 怎么防止越权、误删、危险命令？”

- `defense/失败分类体系与排查话术.md`
  - 用于回答“Agent 失败后怎么定位和迭代？”

- `defense/AI智能体项目面试题库.md`
  - 更完整的题库。

- `defense/面试追问应答策略手册.md`
  - 更长的标准回答手册。

- `defense/代码智能体技术防守说明.md`
  - 项目整体技术答辩笔记。

## 推荐学习顺序

1. `learn/三十分钟面试准备包.md`
2. `demo/五分钟面试演示脚本.md`
3. `learn/核心代码阅读路线图.md`
4. `learn/最近新增Agent能力代码导览.md`
5. `demo/evidence/演示证据目录说明.md`
6. `defense/AI智能体项目面试问答.md` 前 12 问

其余 `defense/` 文件不要当教材顺序读，只当被追问时的补充材料。

## 展示入口

优先使用：

```bash
source .venv/bin/activate
forge ui
```

打开页面后先点：

1. `Show Interview Evidence`
2. `Show Result Summary`
3. `Show Usage Dashboard`
4. `Show Trace Timeline`

CLI 备用：

```bash
forge bench swebench --showcase --agent-mode compare --profile coding_fix --provider deepseek --max-steps 16 --max-revision-rounds 2
forge report latest
```

## 现场展示标准话术

先讲这个项目证明什么：

> 我不是做一个聊天壳子，而是做 coding agent 的 运行时控制平面（runtime control plane）。核心是把 context、tool、policy、trace、usage、eval 这些生产工程问题放进 harness，而不是只靠 prompt。

再讲它不证明什么：

> 当前不能把 patch generated 当成 SWE-bench resolved rate；只有跑官方 harness 后才讲官方结果。我现在展示的是闭环、可观测性、失败归因和 multi-agent artifact（产物） workflow（产物工作流）。

## 一句话定位

> Agent Forge 是一个面向 SWE-bench 的 CodingAgent control plane。它用 AgentLoop 管理可控 runtime，用 MultiAgentCoordinator 组织基于 artifact（产物）交接的 review workflow，并用 SWE-bench compare 与 evidence reports 证明 agent 行为可复现、可审计、可评估。
