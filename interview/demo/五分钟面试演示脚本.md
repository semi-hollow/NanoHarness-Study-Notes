# NanoHarness 五分钟现场演示

目标不是展示功能数量，而是用**一个真实 Run Story + 一次 approval/recovery**证明：你能控制
任务、追踪副作用、读懂 Evidence，并诚实说明边界。

## 面试前准备

提前选择一份可公开、配置身份完整的真实 provider run，确认它至少包含 task/base commit、trace、
patch、local evidence，并明确 official outcome 是 resolved、unresolved 还是 `Unknown`。

```bash
cd /path/to/NanoHarness
source .venv/bin/activate
forge inspect <prepared-real-run>
forge demo
```

Windows 使用 `.venv\Scripts\Activate.ps1`。不要把在线模型稳定性押在现场；真实能力看预生成真实
Run Story，控制面使用可重复 Demo。主演示只依赖 `forge inspect` 和 `forge demo` 两个入口。

## 0:00–0:30：一句话定位

> NanoHarness 是面向真实代码仓库的可治理软件工程 Agent 与评测工作台。它从 issue 和 base
> commit 出发，在隔离 workspace 中运行受治理 AgentLoop；Runtime 管理 Context、工具副作用、
> checkpoint/recovery，Evaluation 严格区分 candidate、local 和 official Evidence。

立即给出输入、输出和问题，不讲包结构，也不说“自研通用框架”。

## 0:30–2:30：一个真实 Run Story

```bash
forge inspect <prepared-real-run>
```

只沿屏幕上的 canonical story 回答：

1. **Task**：issue、base commit、model/runtime/environment identity。
2. **Runtime**：一个关键 Context/Tool 决策及对应 trace event。
3. **Side effect**：写操作经过哪个 policy/operation owner。
4. **Artifact**：candidate patch 的 producer 和 consumer。
5. **Outcome**：local 与 official 分别是什么；缺失时明确说 `Unknown`。

需要细节才展开 raw trace/checkpoint。不要从目录路径、JSON dump 或 CLI 参数开始。

口述桥接：

> 这里展示的是同一份 Run Story Read Model。CLI、默认只读 Workbench 和静态报告不各自推断状态；每个
> artifact 都能返回 producer、source event、consumer，以及它能和不能证明什么。

## 2:30–3:45：一次 Approval / Recovery

```bash
forge demo
```

`forge demo` 会经 `Harness.run` 用一个命令依次完成等待、人工决定和正式 continuation。只指出五个节点：

```text
write ToolCall
  -> operation identity / target fingerprint
  -> waiting_approval durable checkpoint
  -> resume as a new continuation
  -> ledger stale/replay check -> side effect at most once
```

必须主动说明：

- Approval 授权的是具体 operation，不是给 Agent 永久写权限。
- Resume 不恢复 Python stack、HTTP connection 或模型隐藏状态。
- 已执行 operation 由 ledger 防重放；target drift 会令旧批准 stale。
- Deterministic Demo 证明控制面，不证明模型推理质量。

真实任务的 operator 入口是 `forge resume`：`--answer` 回答 clarification，
`--decision approved|rejected` 处理 approval；五分钟内不再启动第二条命令链。

## 3:45–4:30：Evidence 边界

返回真实 Run Story，指向而不是背诵三层 Evidence：

```text
candidate patch -> local verification -> official evaluator outcome
```

> 非空 patch 只证明产生了候选修改；local PASS 只覆盖记录的本地检查；只有 parsed official
> outcome 进入 official denominator。没有 official artifact 时结论是未知，不是 0%，更不是 solved。

失败 run 也可以展示：用 taxonomy 将下一次行动指向 Runtime、模型、环境或 evaluator owner，
这比只挑成功截图更能证明工程能力。

## 4:30–5:00：LangGraph 取舍与收口

> `Harness.run` 是唯一 Runtime Public API，AgentLoop 是内部阶段编排；Port 隔离模型、状态、工具
> 和执行环境等真实边界。我不认为它全面优于 LangGraph。普通业务 workflow 应优先复用成熟
> 框架；这个项目深入验证 Coding Agent 特有的副作用治理、恢复和 Evidence contract，这些能力
> 也可作为 node、middleware、checkpointer 或 service 接入现有框架。

最后只保留一个缺口：真实 official repeated campaign 仍是最高优先级；不能靠更多 UI、Runtime
名词或 deterministic Demo 替代。

## 不进入主演示

- Multi-Agent / Fanout timeline；
- Memory 生命周期和 MCP/Skills；
- Smoke-5 五个 case 逐题信息；
- Campaign 命令、完整指标矩阵；
- TUI、Workbench launcher 和全部 CLI；
- 第二个 HITL/approval 场景。

这些只在面试官追问时从对应 owner 展开。

## 演示失败时

- `inspect` 视图不可用：打开同一 Run Story 的静态报告，不改用另一套状态判断。
- Demo 中断：打开已准备的 checkpoint/operation/continuation story，并说明未现场完成的步骤。
- Official evaluator 缺失：显示 denominator 为 `Unknown`，绝不伪装成 0% 或 resolved。
- 找不到 artifact：停止翻目录，使用 `forge inspect <run-or-artifact>` 查 manifest/catalog。
- 时间不足：保留真实 Run Story 和 Evidence 边界，删掉框架扩展说明。

## 演示后自评

| 检查 | 通过条件 |
| --- | --- |
| 定位 | 30 秒内说清输入、输出、问题和边界 |
| 主线 | 全程只有一个真实 Run Story |
| 控制 | Approval、checkpoint、resume、ledger 没有混讲 |
| 证据 | candidate/local/official 没有混淆，结论能返回 artifact |
| 克制 | 未展开 Multi/Fanout/Memory/MCP，未声称 production/distributed |
| 时间 | 五分钟内结束，追问内容没有提前倾倒 |

任何一项失败，回到[闭卷自测与反馈](../../learning/04-闭卷自测与反馈.md)记录错误，不新增讲稿。
