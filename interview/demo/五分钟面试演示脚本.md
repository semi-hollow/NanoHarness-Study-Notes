# NanoHarness 五分钟现场演示

> 层级：[返回总入口](../../README.md) → A6 主学习链 → 面试交付；顺序只以总入口为准。

目标不是展示功能数量，而是用**真实模型 Run Story + 控制面 continuation + Astropy Evidence**
证明：你能控制执行过程、追踪副作用、读懂证据，并诚实说明边界。三个动作都已编排，不输入
路径、不临场调用 Docker，也不从 Workbench 发起 Agent。

## 面试前准备

在 A1 提前完成 Lab 3 和 Lab 4，确认保存的 Live 与 Astropy Evidence 可公开；API key 和敏感
prompt 不能进入演示材料。脚本会以 Focus 模式隐藏本机 project/status 信息，屏幕共享时不要展开
raw provenance。正式面试先点击：

```bash
scripts/interview_demo.sh --show-live
```

它只重新发布已保存的真实 run 并打开只读 Workbench，不调用模型。若没有可展示的真实 run，
改用默认 Control Plane Demo，并明确它不证明模型质量。

## 0:00–0:30：一句话定位

> NanoHarness 是面向真实代码仓库的可治理软件工程 Agent 与评测工作台。它从 issue 和 base
> commit 出发，在隔离 workspace 中运行受治理 AgentLoop；Runtime 管理 Context、工具副作用、
> checkpoint/recovery，Evaluation 严格区分 candidate、local 和 official Evidence。

立即给出输入、输出和问题，不讲包结构，也不说“自研通用框架”。

## 0:30–2:20：真实模型 Run Story

只沿 Workbench 的同一份 canonical story 回答：

1. **Task**：固定 bug、model/runtime/environment identity。
2. **Context**：模型实际看到的 messages 与 tool schemas。
3. **Action**：ToolCall 怎样经过 routing、guardrail、authorization 后执行。
4. **Artifact**：patch、trace、checkpoint 的 producer/consumer。
5. **Boundary**：这次 focused pytest 或缺失证据具体能证明什么。

口述桥接：

> Debugger 用来掌握活的内存和因果过程；这里的 Workbench 只读回放同一 Runtime 落盘的 Run
> Story。CLI、页面和报告不各自猜状态，artifact 都能返回 producer、source event 与 claim boundary。

不要打开 raw JSON，也不要把 Live fixed repair 说成 official resolved。

## 2:20–3:40：一次 Approval / Recovery

点击第二个已编排动作；Workbench 会切换到刚生成的控制面 Evidence：

```bash
scripts/interview_demo.sh
```

只指出这条状态序列：

```text
write ToolCall
  -> operation identity / target fingerprint
  -> waiting_approval durable checkpoint
  -> explicit decision
  -> new continuation -> side effect at most once
```

必须主动说明：

- Approval 授权具体 operation，不是给 Agent 永久写权限。
- Resume 加载 durable checkpoint，不恢复 Python stack、HTTP connection 或模型隐藏状态。
- operation ledger 防重放；target drift 会令旧批准 stale。
- 确定性 ModelPort 只固定工具意图；控制面仍走正式 `Harness` / `AgentLoop`。
- 这一段证明治理机制，不证明模型推理质量、pytest 或 official resolved。

## 3:40–4:30：Astropy 三层 Evidence

点击第三个动作，只切换已保存的 Lab 4 结果，不现场重跑：

```bash
scripts/interview_demo.sh --show-astropy
```

指向而不是背诵：

```text
candidate patch -> local verification -> official evaluator outcome
```

> 非空 patch 只证明产生候选修改；local PASS 只覆盖记录的本地检查；只有 parsed official
> outcome 进入 official denominator。没有 official artifact 时是 Unknown，不是 0%，更不是 solved。

失败结果也能展示：用 `patch_generated`、`local_verified`、`official_resolved` 定位失败层，而不是
隐藏失败或用模型自述替代 Oracle。

## 4:30–5:00：架构取舍与收口

> `Harness.run` 是唯一 Single-Run Public API，AgentLoop 是内部阶段编排；Port 隔离模型、状态、
> 工具和执行环境等真实边界。我不认为它全面优于 LangGraph。普通业务 workflow 应优先成熟
> 框架；本项目深入验证 Coding Agent 特有的副作用治理、恢复和 Evidence contract。

最后只保留一个高价值缺口：真实 repeated official campaign；不能靠更多 UI、Runtime 名词或
deterministic Demo 替代。

## 不进入主演示

- Multi-Agent / Fanout、Memory、MCP、Skills；
- Smoke-5 五题细节、Campaign 命令和完整指标矩阵；
- SWE-bench Docker、JSON parser、Workbench 前端；
- 全部 CLI 或第二个 HITL 场景。

## 演示失败时

- Live Evidence 不可用：执行默认控制面脚本，明确降低 claim，不临场修环境。
- Workbench 未启动：打开同一 run 的静态报告，不换一套状态判断。
- Official 缺失：显示 `Unknown`，绝不伪装成 0% 或 resolved。
- 时间不足：保留真实 Run Story、控制面和 Evidence 边界，删掉框架扩展说明。
