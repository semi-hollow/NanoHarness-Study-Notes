# Agent Forge 5 分钟面试 Demo 脚本

目标：把项目讲成一个可复现、可审计、可评估的 CodingAgent harness，而不是“我写了几个 agent 互相调用”。

## 0. 面试前准备

优先用浏览器展示：

```bash
cd /Users/chenjiahui/Documents/GitHub/NanoHarness
source .venv/bin/activate
forge ui
```

如果不想开 UI，就走 CLI：

```bash
source .venv/bin/activate
forge doctor
forge bench swebench --showcase --agent-mode compare --profile coding_fix --provider deepseek --max-steps 16 --max-revision-rounds 2
forge report latest
```

如果 Docker 或 `swebench` 缺失，不影响 demo。可以这样说：

> 本地展示重点是 candidate patch、trace、usage、single-vs-multi comparison。official resolved-rate 必须等 SWE-bench Docker harness 跑完才算；本项目不会把 patch_generated 伪装成 resolved。

## 1. 30 秒项目定位

推荐话术：

> Agent Forge 是一个 SWE-bench-oriented CodingAgent control plane。它不是 IDE，也不是 prompt demo。核心是 AgentLoop：负责 context 构建、模型调用、tool routing、安全策略、trace 和 usage 记录。Multi-agent 是外层 coordinator-driven workflow，用 Implementer、Reviewer、Verifier 三个角色，通过 artifact handoff 和 bounded revision loop 做审稿验证。最后用 SWE-bench compare mode，在隔离 workspace 中比较 single-agent 和 multi-agent 的质量信号、成本和失败模式。

不要说：

- “这个项目已经替代 Claude Code / Codex。”
- “multi-agent 一定更好。”
- “patch_generated 就代表问题解决。”

## 2. 5 分钟展示路线

### 第 1 分钟：打开 UI 总览

在 `forge ui` 页面点：

1. `Run Doctor`
2. `Run Reference Case`，或者已有 run 时直接点 `Show Interview Evidence`

讲法：

> 我先不让面试官看 raw JSON，而是把 trace、usage、multi-agent summary、comparison report 整理成展示卡片。真实排障时仍然能回到原始 artifact。

### 第 2 分钟：讲 AgentLoop

打开 `Show Trace Timeline`。

讲法：

> AgentLoop 每一步都会重新组装 context。模型只提出 tool call，runtime 决定是否允许执行。工具结果以 observation 回到下一轮。trace 记录 context、LLM、action、tool observation、guardrail、permission，所以失败后能定位到底是哪一层出问题。

对应代码：

- `agent_forge/runtime/agent_loop.py`
- `agent_forge/context/context_builder.py`
- `agent_forge/tools/tool_router.py`
- `agent_forge/observability/trace.py`

### 第 3 分钟：讲 Multi-Agent

打开 `Show Interview Evidence`，看 `Role Timeline` 和 `Artifact Handoff Graph`。

讲法：

> Multi-agent 没有自由聊天。Coordinator 顺序调用同一个 AgentLoop，但每个 role 都有 RoleSpec、tool allowlist、max steps 和 expected artifact。角色之间只通过落盘 artifact 交接，所以审稿和验证都能复盘。

对应代码：

- `agent_forge/multi_agent/coordinator.py`
- `agent_forge/multi_agent/profiles.py`
- `agent_forge/multi_agent/artifacts.py`

### 第 4 分钟：讲 Evaluation

打开最新 run 的：

```text
.agent_forge/runs/<run-id>/cases/<case>/evaluation_report.md
.agent_forge/runs/<run-id>/cases/<case>/comparison.json
```

讲法：

> compare mode 会为同一个 SWE-bench case 准备 single 和 multi 两个隔离 workspace，避免一个 run 的 patch 污染另一个 run。报告不只看成功率，还看成本、LLM calls、tool calls、failed tools、revision rounds、reviewer findings、verifier status 和 failure taxonomy。

### 第 5 分钟：讲 Safety 和 Failure

打开 `Show Interview Evidence` 的 `Safety & Failure Taxonomy`，或看：

- `docs/technical-defense/defense/Agent安全边界与权限防守说明.md`
- `docs/technical-defense/defense/失败分类体系与排查话术.md`

讲法：

> 生产 Agent 的关键不是让模型更自由，而是把风险放进 runtime：workspace sandbox、command policy、permission hook、read-only reviewer、patch anchor 唯一匹配、max steps、timeout、repeated tool call detection。失败也不只说 failed，而是分成 provider、context、tool/runtime、validation、official eval availability。

## 5. 项目亮点、限制、下一步

亮点：

- 单 Agent runtime 和 Multi-Agent coordinator 分层清楚。
- 通过 SWE-bench compare 做真实 case 证据，不靠玩具 demo。
- trace/usage/evaluation/failure taxonomy 能解释结果。
- 安全边界放在 runtime，而不是只靠 prompt。

限制：

- 本地版不是生产 SaaS。
- multi-agent 当前是 deterministic sequential，不是真并发。
- official SWE-bench evaluation 依赖 Docker/swebench 环境。
- resolved-rate 需要更多 case 和官方评测支撑。

下一步：

- 跑固定 regression set，形成多 case evidence。
- 把 failure taxonomy 回流成 eval set。
- 增加更完整的 write/apply_patch stale precondition。
- 在生产形态中把 coordinator 接到异步队列、隔离 sandbox 和 artifact store。
