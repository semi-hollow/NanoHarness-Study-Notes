# NanoHarness 五分钟面试演示脚本

目标：现场证明 NanoHarness 是可执行、可审计、可评测的 Agent Runtime Control Plane，不靠
历史截图或 raw JSON 讲故事。

## 0. 面试前准备

```bash
cd /Users/chenjiahui/Documents/GitHub/NanoHarness
source .venv/bin/activate
forge doctor
```

另开一个终端备用：

```bash
forge ui
```

不要把 API key 放进页面截图。现场无网络时，Benchmark Inspector、HITL 和 Approval
showcase 仍可确定性演示。

## 1. 第 0:00-0:30：一句话定位

> NanoHarness 是一个面向真实仓库任务的精简 Agent Runtime Control Plane。模型负责提出
> tool intent，Harness 负责 context、工具可见性与执行、安全审批、checkpoint、trace 和
> evaluation；最后用 SWE-bench-shaped loop 判断 Runtime 改动改善了什么，而不是把生成 diff
> 直接说成 solved。

马上补一个具体输入输出：

> 输入是 repository、base commit 和 issue；输出是 candidate patch，以及 trace、usage、local
> validation、official result 和 failure diagnosis。

## 2. 第 0:30-1:40：先展示 Benchmark 契约

执行：

```bash
forge bench cases
```

只讲四件事：

1. 候选全集是 SWE-bench Lite test 的 300 个 case；
2. Smoke-5 是五仓库、五问题族的人工分层机制回归样本；
3. 五题用于低成本发现 Harness regression，不代表总体 resolved rate；
4. 单次运行不能估计随机方差。

再执行：

```bash
forge bench case astropy__astropy-12907
```

指着页面讲：

> 这里是 Agent 真正收到的 issue、固定 base commit、2 个 FAIL_TO_PASS 和 13 个
> PASS_TO_PASS。默认不显示 official test patch 和 gold patch，避免在调试时污染输入；运行后
> 复盘才显式打开。

主要代码：

- `agent_forge/bench/domain/catalog.py`
- `agent_forge/bench/application/case_inspection.py`
- `agent_forge/bench/presentation/case_inspection.py`
- `docs/evaluation/smoke-5-case-catalog.md`

## 3. 第 1:40-2:40：展示 Runtime 控制链

打开代码：

- `agent_forge/runtime/application/agent_loop.py` 的 `AgentLoop.run`
- `agent_forge/runtime/application/turn_preparation.py` 的 `TurnPreparation.execute`
- `agent_forge/runtime/application/tool_execution.py` 的 `ToolExecutionPipeline.execute_calls`
- `agent_forge/runtime/control.py` 的 `StepController`

讲法：

> `AgentLoop.run` 只展示 phase order。每轮先组装受预算约束的 context，再调用模型；模型只提
> tool intent，ToolRouter、registry、permission、command policy 和 sandbox 决定能否执行。
> Single Loop 没有 DAG；max steps、timeout、累计 cost、连续失败、exact repeat 和 tool burst
> 提供硬停机边界。Trace 记录每个决定，失败后能判断是 provider、context、tool、policy 还是
> validation 问题。

不要说“Tool + 入参 + 结果做 hash”。当前 exact repeat identity 是 Tool + canonical arguments。

## 4. 第 2:40-3:50：现场演示 HITL 或审批

### 方案 A：需求澄清

```bash
forge showcase hitl start
```

命令会打印下一条 continuation 命令。执行其中的：

```bash
forge showcase hitl continue <run-dir> --answer "Use the backward-compatible behavior"
```

讲法：

> 固定 model tool call 只负责稳定触发，真正经过的是正式 AgentLoop、HumanInputRepository、
> checkpoint、continuation 和 trace。它证明控制面，不证明模型推理质量。

### 方案 B：写操作审批

```bash
forge showcase approval start
```

先确认 run directory 中目标文件未修改，再按输出执行 continuation。讲清批准前无副作用、批准后
才执行，以及 operation fingerprint/stale check。

主要代码：

- `agent_forge/showcase/control_plane.py`
- `agent_forge/runtime/application/run_lifecycle.py`
- `agent_forge/runtime/application/tool_authorization.py`
- `agent_forge/runtime/application/operator_control.py`

## 5. 第 3:50-5:00：UI 证据与实验边界

打开 `forge ui` 的最新 run，依次看：

1. Result Summary：candidate/local/official 三层状态；
2. Multi-Agent Evidence：role artifact、reviewer/verifier、Single timeline；
3. Usage & Runtime：token、cost、tool failure、context compaction；
4. Safety & Failure：policy、sandbox、failure taxonomy；
5. Artifact 内容页：直接看报告，不要求面试官打开 JSON。

讲法：

> Benchmark 请求会固定 provider/model/temperature、case、环境和预算。Scorecard 比较时拒绝
> 未声明配置漂移；temperature 也进入真实 HTTP 请求和 experiment identity。只有 official
> per-case `resolved=true` 才算 resolved，candidate patch 和 local verified 都不会冒充官方结论。

## 6. 被追问 Multi-Agent 时

先区分两条路径：

| 路径 | 用途 | 真实边界 |
| --- | --- | --- |
| Sequential Coordinator | Implementer -> Reviewer -> Verifier，artifact handoff 和 bounded revision。 | 不是自由聊天 swarm。 |
| Live Fanout | 用户提供经过验证的 task DAG，独立 worktree 并发执行，scope/touched-file/merge gate 和 partial recovery。 | 不做 model-driven 自动拆解，不是 distributed worker service。 |

主要代码：

- `agent_forge/multi_agent/application/coordinator.py`
- `agent_forge/multi_agent/application/live_fanout.py`
- `agent_forge/multi_agent/domain/live.py`

不要用 Fanout DAG 解释 Single Agent 空转。

## 7. 现场不能做的声明

- “Smoke-5 代表 SWE-bench Lite 总体能力。”
- “patch generated 就是 solved。”
- “temperature 为 0 就保证每次完全相同。”
- “项目已经有通用 semantic no-progress detector。”
- “request cancel 会暂停当前 task 或回滚副作用。”
- “multi-agent 一定比 single-agent 好。”
- “这是 production IDE / SaaS / distributed swarm。”

## 8. 十分钟扩展顺序

有更多时间时，再展示：

```bash
forge bench swebench \
  --regression-set smoke-5 \
  --provider deepseek \
  --model deepseek-chat \
  --temperature 0 \
  --tool-routing task-aware \
  --evaluate
```

以及 matched ablation：

```bash
forge eval ablation <control-run> <treatment-run> --factor tool-routing
```

最后一句收口：

> 这个项目最有价值的不是能力数量，而是每个能力都要进入真实 Runtime path、留下机器可读
> evidence，并明确它能证明什么、不能证明什么。
