# NanoHarness 五分钟现场演示

目标不是展示功能数量，而是证明你能控制一次任务、读懂证据并承认边界。

## 面试前检查

```bash
cd /Users/chenjiahui/Documents/GitHub/NanoHarness
source .venv/bin/activate
forge doctor
forge ui
```

另开终端准备：

```bash
forge bench cases --regression-set smoke-5
forge showcase hitl start
```

不要依赖某个难找的历史 trace。现场入口必须可重复；真实模型任务可以使用已公开且配置一致的
run evidence，控制面演示使用 deterministic showcase。

## 0:00–0:30：一句话定位

> Coding Agent 不难生成 patch，难的是让真实仓库里的工具副作用可治理、任务中断后可恢复，
> 并且知道 patch 是否真正解决问题。NanoHarness 从 repository issue 出发，在隔离 workspace
> 中运行受治理 AgentLoop，再用 local/official evaluation 和 repeated campaign 验收结果。

说完立刻指出输入、输出和三层 evidence，不讲包结构。

## 0:30–1:20：展示一个任务与证据

在 Workbench 的 `Run Evidence` 中只回答五个问题：

1. issue 和 base commit 是什么？
2. 使用了哪个 model/runtime preset？
3. Agent 是否产生 candidate patch？
4. local 与 official outcome 分别是什么？
5. 失败时 taxonomy 指向哪个 owner？

不要从 absolute path 或 raw JSON 开始。需要细节时再展开 trace、tool timeline 和 artifact。

## 1:20–2:10：展示 Benchmark 契约

```bash
forge bench cases --regression-set smoke-5
forge bench case astropy__astropy-12907
```

必须主动说：候选全集是 SWE-bench Lite test 300；Smoke-5 是五仓库、五问题族的低成本机制
回归集；每题有 F2P/P2P；运行前隐藏 test/gold patch；五题不能代表总体 solved rate。

## 2:10–3:20：现场 HITL 或审批

方案 A：

```bash
forge showcase hitl start
```

指出 pending question、`waiting_human` checkpoint 和 continuation command。说明回答只补信息，
不授权写操作。

方案 B：

```bash
forge showcase approval start
```

指出 operation fingerprint、pending approval、resume 和最终副作用。说明 target drift 会使旧
approval stale，已执行副作用不会被假装回滚。

只演示一个方案；另一个用于追问。

## 3:20–4:15：展示 Multi-Agent 边界

在 UI 中先展示 Multi timeline，再展示 Single timeline。回答：

- 顺序 coordinator 通过 artifact handoff 协作。
- Live fanout 在 disposable worktree 中运行显式 DAG worker。
- 声明 scope 重叠会串行，scope escape 或 patch conflict fail closed。
- 当前是本机 coordinator，不是 distributed swarm，也不自动拆任务。

不要声称 Multi-Agent 更强。只有 matched official outcome、成本和稳定性才能回答是否值得。

## 4:15–5:00：框架选择与收口

> `Harness.run` 是稳定接入面，AgentLoop 是内部 kernel，Policy、State、Environment 和
> Evaluation 通过显式 Port 组合。我不认为它全面优于 LangGraph；普通业务 workflow 会优先
> 复用成熟框架。这个项目的价值是完整实现并验证 Coding Agent 特有的副作用、隔离和证据
> 边界，这些组件也可以作为 node、middleware 或 service 接入现有框架。

最后只保留一个未完成点：真实 official repeated campaign 结果仍是当前最高优先级，不再继续
堆 Runtime 名词。

## 演示失败时

- UI 不可用：用 `forge report latest`，不要现场调前端。
- 在线模型不稳定：切换 deterministic showcase，明确它只证明控制面。
- Official evaluator 缺失：显示 denominator 为 unknown，不展示伪 0%。
- 找不到 artifact：停止翻目录，改用 `forge replay latest` 或 Workbench catalog。

## 演示后自评

| 检查 | 通过条件 |
| --- | --- |
| 定位 | 30 秒内说清输入、输出、问题和边界 |
| 证据 | candidate/local/official 没有混淆 |
| 操作 | 命令无需临时查文档 |
| 代码 | 能指向 `Harness.run`、`AgentLoop.run`、工具管线和 lifecycle |
| 克制 | 没有声称 distributed、production-ready 或总体 solved rate |
| 时间 | 5 分钟内结束，追问内容未提前倾倒 |

任一项失败，回到闭卷自测对应题，不通过新增讲稿解决。
