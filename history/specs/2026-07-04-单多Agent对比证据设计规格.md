# Compare Evidence Design

Date: 2026-07-04

## Purpose

NanoHarness / Agent Forge should feel like an interview-ready AI Agent engineering project, not a toy multi-agent demo. The next slice finishes the most important credibility gap from the original plan: an honest single-vs-multi comparison flow that runs the same SWE-bench-shaped case through isolated single-agent and multi-agent workspaces, then writes evidence artifacts that explain quality signals, cost, tool behavior, revision behavior, and caveats.

The goal is not to claim that multi-agent is globally better. The goal is to make the project able to say, with artifacts: when multi-agent adds useful review/verification signal, and when single-agent is cheaper or sufficient.

## Scope

Implement a minimal but real compare evidence path:

- `forge bench swebench --agent-mode compare` no longer raises `NotImplementedError`.
- The same case is run twice: once as `single`, once as `multi`.
- Runs use isolated per-variant workspaces and artifact directories.
- Existing `AgentLoop` and `MultiAgentCoordinator` remain the execution engines.
- Evaluation helpers extract comparable metrics from existing run artifacts.
- Each compared case writes `comparison.json` and `evaluation_report.md`.
- The benchmark run report remains honest about official SWE-bench evaluation being unavailable/skipped when Docker or `swebench` is missing.

## Non-goals

- Do not implement parallel multi-agent execution.
- Do not introduce swarm/quorum/consensus behavior.
- Do not add Claude provider support.
- Do not add large UI rewrites or new heavy dependencies.
- Do not require official SWE-bench Docker evaluation for local tests.
- Do not tune `research_report` deeply in this slice.
- Do not claim resolved-rate without official evaluation.

## Architecture

The compare path stays as an outer benchmark orchestration layer:

```text
forge bench swebench --agent-mode compare
        |
        v
SWE-bench case loader
        |
        v
prepare isolated workspaces
  - cases/<case>/single/workspace
  - cases/<case>/multi/workspace
        |
        v
single: AgentLoop
multi : MultiAgentCoordinator(coding_fix)
        |
        v
collect existing artifacts
  - trace.json
  - usage.json / usage_report.md
  - patch.diff
  - result metadata
  - multi_agent/multi_agent_summary.json
        |
        v
EvaluationComparison
        |
        v
comparison.json + evaluation_report.md
```

`agent_forge/bench/swebench.py` owns case orchestration and workspace isolation. `agent_forge/evaluation/` owns metric normalization and report rendering. Runtime internals remain unchanged.

## Components

### SWE-bench compare runner

`agent_forge/bench/swebench.py` will replace the current compare `NotImplementedError` with a real loop. For each selected case:

1. Prepare a clean `single` workspace variant.
2. Run the existing single-agent path.
3. Prepare a clean `multi` workspace variant.
4. Run the existing multi-agent path with the selected profile and revision budget.
5. Write variant-specific patches and metadata.
6. Call evaluation comparison helpers to write per-case comparison artifacts.

The two variants must not share a mutable checkout. This matters more than performance because interview credibility depends on fair comparison.

### Metric extraction

Add a small helper in `agent_forge/evaluation/`, likely `metrics.py`, that reads dictionaries loaded from existing artifacts and returns flat metrics accepted by `compare_runs()`.

First-version metrics:

- status
- patch_generated
- official_eval_status
- cost_usd
- llm_calls
- tool_calls
- failed_tool_calls
- revision_rounds
- verifier_status
- reviewer_findings
- failure_taxonomy

Missing data should become `0`, `False`, `None`, or `unavailable`, never a crash and never a fake success.

### Evaluation report

Improve `agent_forge/evaluation/report.py` from a field dump into an interview-readable evidence card:

1. Executive summary
2. Side-by-side metrics table
3. Multi-agent review loop summary
4. Failure taxonomy
5. Recommendation and caveats

The report should preserve the key principle: compare quality, cost, and failure mode case by case.

### TUI compatibility

If the TUI constructs a SWE-bench namespace without new fields required by `run_swebench_from_args()`, add defaults:

- `agent_mode='single'`
- `profile='coding_fix'`
- `max_revision_rounds=2`

This is a small reliability fix, not a UI redesign.

## Data flow

For a compared case, the benchmark output should be shaped like:

```text
.agent_forge/bench/<run-id>/
  cases/<case-id>/
    single/
      trace.json
      patch.diff
      usage_report.md
      result.json or metadata equivalent
    multi/
      trace.json
      patch.diff
      usage_report.md
      multi_agent/
        multi_agent_summary.json
        multi_agent_report.md
    comparison.json
    evaluation_report.md
  report.md
  predictions.jsonl
```

Exact filenames may follow existing conventions where they already differ; the important invariant is that single, multi, and comparison artifacts are discoverable and separated.

## Error handling and honesty

- If a variant fails, compare should still write a comparison artifact with that variant marked failed/blocked where possible.
- If official SWE-bench evaluation is not run, report `skipped` or `unavailable` with the reason.
- If a metric cannot be extracted, report it as unavailable/default rather than inventing values.
- If multi-agent ends with `needs_revision`, that is a legitimate outcome and should be shown, not hidden.
- If Docker or `swebench` is missing, the benchmark report must keep saying generated patches are not resolved-rate claims.

## Testing

Add or extend focused tests instead of running real Docker SWE-bench:

1. `tests/test_swebench_compare.py`
   - compare mode runs without `NotImplementedError` using local/fake case data.
   - single and multi variants use separate output/workspace paths.
   - per-case `comparison.json` and `evaluation_report.md` are written.

2. `tests/test_evaluation_comparison.py`
   - metric extraction handles nested usage/multi-agent summaries.
   - missing fields do not crash or fabricate success.
   - report includes executive summary, side-by-side metrics, and recommendation.

3. Existing test suite remains green.

Verification commands:

```bash
source .venv/bin/activate && python -m compileall -q agent_forge tests
source .venv/bin/activate && python -m unittest discover tests
bash scripts/verify.sh
git diff --check
```

## Interview positioning

After this slice, the project can be explained as:

> Agent Forge is not a swarm demo. It is a reproducible coding-agent control plane with explicit evidence. Multi-agent is an outer artifact-based workflow, and its value is measured case by case against a single-agent baseline using isolated workspaces, common metrics, and honest evaluation caveats.

This directly serves the user goal: high information density, compact code, low learning cost, strong visual/report evidence, and no unnecessary architecture bloat.
