# Interview-Oriented Agent Harness Design

**Goal:** Turn NanoHarness from a runnable agent demo into a compact, interview-defensible AI Agent systems workbench: observable, evaluable, safe, explainable, and honest about trade-offs.

## User intent and success criteria

The project is primarily for preparing and passing 3+ years AI Agent developer interviews. It should not optimize for the strongest possible agent at all costs. It should optimize for:

- **High interview signal:** show production-ish judgment: orchestration, safety boundaries, evals, traces, cost, failure analysis, and isolation.
- **Low learning cost:** fewer duplicated code paths, small focused modules, high-density docs, and visual explanations that make the core architecture easy to explain.
- **Honesty:** do not claim true distributed/swarm production deployment if this repo implements a local/coordinator harness. Instead, explicitly present it as a parallel-ready teaching and evaluation harness with a clear production evolution path.
- **Visual leverage:** richer dashboard/reporting is allowed even if frontend code is generated and not deeply studied, because visuals help the user understand and present the system faster.

## Architecture

NanoHarness should be framed as an **Agent systems experiment bench**. The core runtime remains a controlled coordinator + agent loop, not an uncontrolled swarm. The system records traces, artifacts, usage, tool decisions, failures, and evaluation outcomes, then renders them into reports and dashboards.

The implementation should prefer small Python modules over large framework additions:

- `agent_forge/evaluation/extractors.py` normalizes metrics from traces, usage summaries, bench results, and multi-agent summaries.
- `agent_forge/evaluation/reporting.py` renders Markdown/HTML comparison reports and interview-friendly visuals.
- Existing `bench/report.py` and `bench/swebench.py` call those shared helpers instead of duplicating parsing logic.
- Multi-agent code keeps artifact handoff and role isolation explicit, with clear status preservation and tool matrices.

## Visual design

Reports may include richer dependency-free HTML/CSS/SVG. No server or React app is required. The visual package should include:

1. Single vs multi comparison dashboard.
2. Role swimlane showing coordinator-controlled stages and revision rounds.
3. Artifact DAG / handoff chain.
4. Failure taxonomy matrix.
5. Role/tool safety matrix.
6. Cost/usage summary and recommendation panel.

Charts must be accessible: table fallback, labels not color-only, dark-mode-friendly surfaces, no dual-axis charts, and categorical colors used consistently.

## Multi-agent concurrency positioning

The repo must avoid fake claims. The preferred interview framing is:

> This project implements a deterministic local coordinator with artifact-based handoffs and bounded revision loops. It is parallel-ready by design: independent role calls can be executed concurrently behind the same artifact/status interfaces, but the default local runner prioritizes reproducibility, debuggability, and learning cost over raw throughput.

The docs should include a production evolution section:

- local sequential coordinator today;
- async task executor next;
- worker pool / queue / managed sandbox for production;
- per-agent isolated workspaces if agents mutate files in parallel;
- shared artifact store and trace IDs for joins;
- cost/latency trade-offs and failure handling.

It is acceptable to include demo/teaching visuals for parallel scenarios, but they must be labeled as **parallel-ready design / simulated timeline** unless backed by actual concurrent execution.

## Safety and governance

The project should show that agent power is controlled:

- role-specific tool allowlists;
- reviewer read-only by default;
- verifier command access separated from write access;
- missing tools are not advertised in prompts;
- dangerous shell/write actions are explicit and traceable;
- workspace isolation for single-vs-multi comparison;
- honest skipped/unavailable official eval states.

## Evaluation and reporting

Reports should answer: when is multi-agent worth it?

Metrics must be extracted through one normalized path:

- patch generated;
- official eval status: ran/pass/fail/skipped/unavailable;
- tokens and cost if available;
- LLM calls;
- tool failures;
- revision rounds;
- final verdict/recommendation;
- failure taxonomy counts.

Recommendation logic must not blindly prefer multi-agent. It should weigh quality, eval status, patch presence, failures, and cost. If evidence is unavailable, say so.

## Documentation deliverables

Update docs so the user can learn and defend the project quickly:

- README positioning: not a toy agent, but an experiment bench.
- `docs/多Agent协作机制与对比评测说明.md`: coordinator, roles, artifacts, safety matrix, parallel-ready evolution.
- `docs/evaluation/评测目录说明与SWE-bench使用入口.md`: how to run and read comparison artifacts.
- `docs/interview/agent-forge-defense-zh.md`: Chinese interview defense guide.
- `.agent_forge/codex_handoff.md`: factual completion/skipped/verified status.

## Scope boundaries

Do not add heavy dependencies, a persistent web server, Raft/swarm/quorum complexity, or fake official eval pass states. Prefer clear static artifacts and testable Python helpers.

## Self-review

- No placeholders: this spec defines concrete module, report, test, and documentation targets.
- No contradiction: rich static frontend is allowed; heavy framework/service is not required.
- Scope is broad but coherent: one implementation plan can sequence correctness, reports, docs, and verification because they all support the same evaluation/reporting axis.
